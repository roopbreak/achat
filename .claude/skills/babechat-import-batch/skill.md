# Babechat Import Batch — 다중 캐릭터 순차 이식 오케스트레이터

베이비챗 캐릭터 **여러 개**를 한 번에 받아 AChat 스토리로 일괄 이식한다. 단일 캐릭터 절차([`babechat-import`](../babechat-import/skill.md))를 **한 캐릭터씩 완전 순차로** 실행하되, 영속 원장(ledger)으로 **모든 단계·게이트의 수행을 강제**해 절차 누락을 차단하는 것이 이 스킬의 유일한 존재 이유다.

> 이 스킬은 단일 절차를 **복제하지 않는다**. 각 단계의 실제 작업 정의·파싱 규칙·API 스펙·게이트 문구는 전부 `babechat-import/skill.md`가 정본이다. 이 스킬은 그 위에 **순서 강제 + 누락 차단 + 진행 추적** 레이어만 얹는다. 두 파일이 어긋나면 단일 스킬이 우선한다.

---

## 0. 왜 이 스킬이 필요한가 (절대 잊지 말 것)

여러 캐릭터를 "병렬"로 머릿속에 올려두고 처리하면 — 한 캐릭터의 수집을 하다 다른 캐릭터의 다운로드로 점프하고, 게이트 승인을 묶어서 건너뛰고, 검증 스모크를 빠뜨리는 — **절차 누락 사고가 반복적으로 발생**한다. 메모리 `feedback_never_skip_procedure`(절차 생략 절대 금지)와 정면 충돌하는 패턴이다.

**근본 해법은 단 하나: 한 번에 한 캐릭터만 머릿속에 둔다.** 캐릭터 N의 단계 5(검증)까지 원장이 100% ✅가 되기 전에는 캐릭터 N+1의 단계 1을 **절대 시작하지 않는다.** 이 불변식이 깨지는 순간 이 스킬은 의미가 없다.

---

## 1. 핵심 불변식 (INVARIANTS — 위반 금지)

1. **완전 순차**: 동시에 진행 중(in-progress)인 캐릭터는 **항상 정확히 1개**. 캐릭터 N이 `DONE`이 되기 전에 캐릭터 N+1을 손대지 않는다.
2. **원장 우선(ledger-first)**: 어떤 단계든 **작업 전에 원장의 해당 칸을 `▶진행`으로**, **작업 직후 증거와 함께 `✅완료`로** 갱신한다. 원장 갱신 없이 다음 단계로 넘어가지 않는다.
3. **게이트 불가침**: 단일 스킬의 모든 승인 게이트(G1 수집 / G2 다운로드 / G3 매핑표 / 등록 dry-run)를 **캐릭터마다 전부** 거친다. 배치라는 이유로 묶거나 건너뛰지 않는다. (사용자 확정 2026-06-10: 캐릭터별 전체 게이트 유지)
4. **증거 없는 ✅ 금지**: 각 ✅는 검증 가능한 증거(파일 경로·HTTP 상태·장수·응답 saved/skipped 값)를 원장에 동반한다. "했음"이라는 주장만으로 ✅ 불가.
5. **단계 건너뛰기 절대 금지**: 어떤 단계도 "자명해 보여서" 생략하지 않는다. 해당 캐릭터에 진짜 해당 없는 단계(예: 이미지 매핑이 아예 없어 단계 2 N/A)는 생략이 아니라 **원장에 `N/A + 사유`로 명시 기록**한다.

## 2. 입력

```
/babechat-import-batch <url1> <url2> <url3> ...
또는 줄바꿈으로 나열된 URL 목록
```

- 각 URL에서 characterId 추출(`/character/u/{id}/` 또는 `/character/edit/{id}`)
- slug: 미지정 시 캐릭터명 기반 영문 slug를 캐릭터마다 자동 제안(단계 1 게이트에서 사용자 확인). `--slug` 인라인 지정은 단일 URL일 때만 의미 있음.
- **각 캐릭터는 독립 스토리(독립 slug)** 다 → `import/images`의 `charDir=''` 고정 제약은 배치에서 문제되지 않는다(스토리별 slug가 네임스페이스). 단일 스킬의 "멀티 캐릭터 중단" 경고는 *한 스토리에 여러 캐릭터를 욱여넣을 때*만 해당하며, 배치는 그 케이스가 아니다.
- 토큰: **BABECHAT_TOKEN 불필요** — 공개 API/DOM 폴백으로 수집 가능(메모리 `project_babechat_import_ops`). 토큰 요구 문구가 단일 스킬에 있으나 실제로는 무인증 진행.

### 운영 사실 (메모리 `project_babechat_import_ops` 요약 — 배치에서 매 캐릭터 적용)

- 댓글 API 무인증: `https://api.babechatapi.com/ko/api/comments?characterId={id}&sort=likes`. 작가 대댓글: `/comments/{commentId}/replies`. 성인 캐릭터 상세는 safety 차단 → 브라우저 DOM(`get_page_text`) 폴백.
- 이미지 업로드는 **`tmp/babechat-import/upload-images.mjs`(Node)** 로 15장 청크 + 재시도. **zsh bash 루프 금지**(1-인덱스 배열로 청크 깨짐).
- 등록: `node scripts/register-from-md.mjs {dir} --slug {slug} --secret achat2026` (원격 https://risu.ddsmdy.com). 첫 인자는 디렉토리명만.

## 3. 배치 시작 — 원장 생성 (캐릭터 루프 진입 전 반드시)

3-1. URL 전부 파싱 → 캐릭터 목록 확정. 사용자에게 **배치 매니페스트** 제시:

```
배치 대상 N개:
1. {characterId}  추정 slug: {slug1}  — {URL}
2. {characterId}  추정 slug: {slug2}  — {URL}
...
처리 순서: 위 번호순, 캐릭터별 완전 순차(이전 완료 후 다음 시작)
```

→ **배치 승인 게이트(G0)**: 목록·순서·slug 확인. (이 G0는 배치 전체에 1회. 단계별 게이트 G1~등록은 캐릭터마다 별도 유지.)

3-2. 원장 파일 생성: `tmp/babechat-import/_batch/ledger.md` (기존 진행 중 배치가 있으면 §7 재개 참조). 아래 템플릿을 캐릭터 수만큼 채운다:

```markdown
# Babechat Import Batch Ledger
> 시작: {YYYY-MM-DD} | 대상 N개 | 순서: 캐릭터별 완전 순차

## 진행 표 (한 줄 = 한 캐릭터)
| # | slug | S1수집 | S2다운 | S3재작성 | S4매핑 | S4등록 | S5검증 | 상태 |
|---|------|--------|--------|----------|--------|--------|--------|------|
| 1 | {slug1} | ⬜ | ⬜ | ⬜ | ⬜ | ⬜ | ⬜ | 대기 |
| 2 | {slug2} | ⬜ | ⬜ | ⬜ | ⬜ | ⬜ | ⬜ | 대기 |

## 캐릭터별 상세 원장

### [1] {slug1}  — 상태: 대기
- [ ] **S1 수집** (G1) — 캐릭터 상세/DOM, 작가 댓글, 00_source.md
      증거:
- [ ] **S2 이미지 다운로드** (G2) — 매핑 파싱 → 다운 → file 검증
      증거:
- [ ] **S3 스토리 재작성** — create-story 2~3 위임 + 기계적 체크(이중괄호/외부이미지md/자수/상시로어/dry-run)
      증거:
- [ ] **S4-A scene_key 매핑표** (G3) — 04_image_map.md
      증거:
- [ ] **S4-B 등록** (dry-run 게이트) — 사전존재확인 → register → import/images
      증거:
- [ ] **S5 검증** — stories/lore/images GET + 채팅 스모크(이미지 삽입·로어 트리거)
      증거:

### [2] {slug2}  — 상태: 대기
( 동일 6단계 )
```

3-3. **TodoWrite 미러**: 위 단계들을 TodoWrite로도 등록(캐릭터 1의 6단계만 먼저 in/pending; 나머지 캐릭터는 한 줄씩). 원장이 정본, TodoWrite는 시각 보조. 둘 중 하나만 갱신하는 일 없게 매 단계 같이 갱신.

> 원장 생성 없이 캐릭터 처리에 들어가는 것은 이 스킬 위반이다. 원장이 곧 "누락 차단" 장치다.

## 4. 캐릭터 루프 (i = 1 → N, 엄격 순차)

각 캐릭터 i에 대해 아래를 **순서대로** 수행한다. 진입 시 원장 진행 표의 상태를 `진행`으로, 상세 원장 헤더를 `상태: 진행`으로 바꾼다.

> **루프 가드**: 캐릭터 i를 시작하기 전, 원장에서 캐릭터 i-1의 6단계가 전부 ✅(또는 명시적 N/A)이고 상태가 `DONE`인지 **반드시 눈으로 확인**한다. 아니면 i-1로 돌아가 마저 끝낸다. (i=1은 가드 통과)

### S1 — 수집 → 게이트 G1
1. 원장 S1 칸을 `▶진행`.
2. 단일 스킬 **단계 1** 수행: 캐릭터 상세(성인=DOM 폴백) + 작가 댓글(`sort=likes` 필수, `isCreator`/`isPinned`) → `docs/stories/{slug_i}/00_source.md` 저장.
3. **G1 게이트**: 수집 요약(캐릭터명/컨셉 1줄/작가 댓글 수/**이미지 매핑 발견 여부·추정 장수**) 제시 → 사용자 승인.
4. 승인 후 원장 S1 ✅ + 증거(00_source.md 경로, 댓글 수, 매핑 추정). 진행 표 갱신.

### S2 — 이미지 다운로드 → 게이트 G2
1. 원장 S2 `▶진행`.
2. 단일 스킬 **단계 2**: 매핑 위치 탐색(프로필 본문→크리에이터 댓글→외부 갤러리 HTML 해석) → 파싱(항목명 보존) → `tmp/babechat-import/{slug_i}/{카테고리}/`에 **Referer 헤더로** 다운로드 → `file`로 타입/빈파일/장수 검증.
   - 매핑이 어디에도 없으면: 사용자에게 ① 텍스트만 이식 ② NAI 생성 경로 중 선택. 이미지 단계를 조용히 스킵하지 말 것 — 원장에 `S2 N/A(매핑 없음, 사용자 선택: 텍스트만)`으로 기록.
3. **G2 게이트**: 카테고리별 다운로드 결과 표 제시 → 승인.
4. 원장 S2 ✅ + 증거(카테고리별 장수/용량, 검증 통과).

### S3 — 스토리 재작성 (create-story 2~3 위임)
1. 원장 S3 `▶진행`.
2. `00_source.md` → `00_input.md`(외부 이미지 확보·NAI 불필요 명시) → **create-story 단계 2~3만** 위임(persona-codex 컨셉→프롬프트+로어북). **단계 4 풀 QA·단계 5 composition은 생략**(단일 스킬 단계 3 정책 그대로).
3. **기계적 체크(필수, 누락 금지)**: 이중괄호 오타 / `description·first_mes에 외부 이미지 마크다운 `![](http...)` 잔존 여부(있으면 카탈로그 통째 비활성화됨 — 제거) / 자수 지정 / 상시 로어 수 / `register-from-md --dry-run` 통과.
4. 원장 S3 ✅ + 증거(02_prompt.md/03_lorebook 경로, 기계적 체크 5항목 결과).

### S4-A — scene_key 매핑표 → 게이트 G3
1. 원장 S4매핑 `▶진행`.
2. 단일 스킬 **단계 4-A**: 항목명 → 의미 있는 `{카테고리}-{의미}` 영문 scene_key(`[a-zA-Z0-9_-]` ≤80자) → `docs/stories/{slug_i}/04_image_map.md`. 02_prompt.md 이미지 출력 어휘와 일치 확인.
3. **G3 게이트**: 매핑표(원본 항목명 ↔ scene_key ↔ 파일) 제시 → 승인.
4. 원장 S4매핑 ✅ + 증거(04_image_map.md 경로, scene_key 수).

### S4-B — 등록 (dry-run 게이트)
1. 원장 S4등록 `▶진행`.
2. **사전 존재 확인(필수·비가역 지점)**: `GET /api/admin/stories/{slug_i}` + `.../images`. **이미 존재하면 중단·보고** — `story_images`는 append-only라 재실행 시 중복 누적. 이 append가 이 스킬에서 가장 비가역적.
3. 등록: `node scripts/register-from-md.mjs {slug_i} --slug {slug_i} --dry-run` → 확인 → 실등록. (첫 인자 디렉토리명만. `import/card`·`import/zip` 경로 금지 — NAI 자동 트리거.)
4. 이미지 업로드: 파일명 `batch_{sceneKey}_{n}.png|jpg|webp`로 리네임(jpeg/gif는 조용히 skip) → **`tmp/babechat-import/upload-images.mjs`(Node, 15장 청크·재시도)** 로 `POST /api/admin/import/images`(필드명 `images`). 80장/배치(413 회피, 1MB+ PNG는 `sips`로 jpeg 변환). 응답 `saved`/`skipped`가 기대 장수와 일치 확인. 재업로드는 교체 아님 — 교체 시 DELETE 후 업로드.
5. 원장 S4등록 ✅ + 증거(register 응답, import/images saved/skipped, 사전존재확인 결과).

### S5 — 검증
1. 원장 S5 `▶진행`.
2. 단일 스킬 **단계 5**: `GET stories`(필드) + `GET lore`(개수·keys) + `GET images`(수 = 다운 수) + **채팅 스모크**(새 세션→first_mes→1~2턴→`![](/images/{slug_i}/{scene_key})` 삽입·로어 트리거 확인).
3. 스모크에서 회귀 발견 시 그 자리에서 수정 사이클(S3로 부분 복귀) — 다음 캐릭터로 넘어가지 않는다.
4. 원장 S5 ✅ + 증거(각 GET 결과 수치, 스모크 이미지 삽입 여부).

### 캐릭터 i 마감
- 원장 진행 표의 캐릭터 i 6칸이 전부 ✅(또는 N/A)인지 확인 → 상태를 **`DONE`** 으로.
- 사용자에게 **캐릭터 i 완료 1줄 요약**(slug / 로어 수 / 이미지 수 / 스모크 OK) 보고.
- **그 다음에야** i+1로 진입(루프 가드 재적용).

## 5. 게이트 요약 (캐릭터마다 전부)

| 게이트 | 시점 | 확인 내용 |
|--------|------|-----------|
| G0 | 배치 1회 | 대상 목록·순서·slug |
| G1 | 캐릭터별 S1 후 | 수집 요약·이미지 매핑 발견 여부 |
| G2 | 캐릭터별 S2 후 | 다운로드 결과 표 |
| G3 | 캐릭터별 S4-A 후 | scene_key 매핑표 |
| dry-run | 캐릭터별 S4-B 중 | register-from-md dry-run + 사전 존재 확인 |

배치라고 G1~dry-run을 묶거나 자동 통과시키지 않는다.

## 6. 완료 보고 (배치 종료)

모든 캐릭터 `DONE` 후:
1. 원장 진행 표 전체(전부 ✅) 제시.
2. 배치 결과 표: `| slug | 로어 수 | 이미지 수 | 스모크 | URL |`.
3. N/A·이슈가 있었던 캐릭터는 사유 명시.
4. 원장 파일 경로 안내(`tmp/babechat-import/_batch/ledger.md`) — 감사 추적 보존.

## 7. 중단·재개·에러

- **재개**: `tmp/babechat-import/_batch/ledger.md`가 존재하면 새 배치 시작 전 읽는다. 진행 표에서 `DONE`이 아닌 첫 캐릭터의 첫 비-✅ 단계부터 이어간다. 이미 ✅인 단계는 증거를 신뢰하되, S4-B(등록)는 **항상 사전 존재 확인을 다시** 수행(append 중복 방지).
- **한 캐릭터 실패 시**: 해당 캐릭터에서 멈추고 보고. **다음 캐릭터로 자동 진행 금지**(순차 불변식). 사용자가 "이 캐릭터 건너뛰고 다음" 명시하면 원장에 `상태: SKIPPED + 사유` 기록 후 다음으로.
- **에러 대처**: 단일 스킬의 「에러 핸들링」 표가 정본(401/매핑없음/SVG다운/skipped>0/slug 409/멀티캐릭터). 배치에서도 동일 적용하되, 모든 에러는 원장 해당 칸에 기록.
