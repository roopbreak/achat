# AChat 프롬프트 가이드라인 레퍼런스

## 프롬프트 작성 4원칙

### 원칙 1: 모호함 대신 구체적으로
| 모호한 표현 | 구체적 대체 |
|------------|-----------|
| "아름답고 느낌 있는 문체" | "3인칭 전지적 시점으로, 캐릭터의 내면 심리를 중심으로 서술" |
| "적당히 길게" | "캐릭터의 동작과 표정, 시선, 자세를 구체적 디테일로 묘사" |
| "재미있게" | "은유법을 사용하되, 형이상학적 표현은 피하라" |

### 원칙 2: 정보와 지시 분리
```
[캐릭터 정보]
- 성격: 까칠한 CS 담당자
- 말투: 직설적, 반말

[출력 지시]
- 3인칭 시점으로 서술
- 상황 묘사 후 캐릭터 대사를 " " 안에 출력
```

### 원칙 3: 과도한 지시 회피
- 체력/마나 실시간 계산 + 이미지 + 대사 동시 관리 X
- 동시 시스템 2개 이내 권장

### 원칙 4: 구조화
- 마크다운 사용 (##, -, **)
- JSON 사용 금지

## AChat 엔진 호환 규칙

### 시스템 충돌 방지
- 출력 분량 직접 지정 금지 ("500자 이상" 등) → 시스템이 maxTokens로 제어
- 이미지 키워드 방식 금지 → AChat은 scene_key + story_images 테이블 사용
- description에 자주 변하는 내용 넣지 않기 → prompt-caching 효율 저하

### 컨텍스트 빌드 이해 (v2)
```
[캐시됨] system block 1: 서술 규칙 (NARRATION_RULES)
[캐시됨] system block 2: 캐릭터 정보 (description + personality + scenario + 페르소나)
[캐시됨] system block 3: 상시 로어북 (constant=1) + 이미지 카탈로그
[매턴 갱신] system block 4: 유저 노트
[매턴 갱신] system block 5: 동적 컨텍스트 (키워드 로어 + HypaMemory + 세션 요약)
[매턴 갱신] system block 6: post_history_instructions (매 턴 규칙 재주입)
[매턴 갱신] 최근 N턴 대화 (토큰 안전장치로 동적 조절, 최소 5턴 보장)
[매턴 갱신] 현재 유저 입력
```

→ Block 1~3은 캐시되므로 안정적인 내용만
→ 유저 노트 변경 시에도 Block 1~3 캐시 유지
→ 동적으로 변해야 하는 정보는 로어북에
→ post_history_instructions로 매 턴 핵심 규칙 재확인

### 로어북 최적화
- **상시 로어 (constant=1)**: 항상 주입. 수치 규칙, 핵심 세계관. 캐시됨
- **키워드 로어**: 엔트리별 scan_depth(기본 4턴) + 현재 입력에서 매칭
  - AND 조건: `"키1+키2"` → 두 키워드 모두 포함 시 매칭
  - NOT 조건: `"-제외어"` → 해당 단어 포함 시 매칭 차단
  - 명령어 트리거: `"!command"` → 유저가 `!command` 입력 시 활성화 (scan_depth=1 권장)
- **토큰 예산**: 키워드 로어는 예산 내에서만 활성. 우선순위 높은 것부터
- **scan_depth**: 기본 4, 명령어 트리거는 1, 장기 맥락 로어는 8~10

### post_history_instructions (v2 신규)
- 매 턴 system block 마지막에 주입되는 지시사항
- 캐릭터 상태 연속성, 상시 규칙 참조, 스테이터스 출력 등 핵심 지시
- description에 넣기엔 동적이고, 로어북에 넣기엔 항상 필요한 규칙용
- 짧고 핵심적으로 (3~5줄 이내 권장)

### 변수 사용
- `{{char}}`: 이중 중괄호 필수. char_name으로 치환
- `{{user}}`: 이중 중괄호 필수. 페르소나 이름으로 치환
- description/로어북에서는 캐릭터 실제 이름을 직접 사용하는 것이 안전

### 토큰 안전장치 (v2 신규)
- 컨텍스트가 모델 한도(200K)의 85%를 초과하면 RECENT_TURNS 자동 축소
- 최소 5턴은 보장되므로 description/로어북이 극단적으로 길면 최근 대화가 줄어듦
- description은 3,000자 이내, 로어북 총량은 적정선 유지 권장

## 스테이터스 시스템

### 기본 4종 (권장)
```
📍 장소: (현재 장소)
👗 복장: (착장 + 상태)
🎬 상황: (장면 맥락)
💭 속마음: "(내면 독백)"
```

### 수치 스탯 (선택)
- 1000 스케일 사용 (0~100은 AI가 급변시킴)
- "화폐" 개념으로 선언 → AI가 신중하게 변동
- 평시 ±10~30, 극적 이벤트 ±50, 절대 ±100 이상 금지

### 선택지 (선택)
- ①② 상황 맞는 행동 (서술체, 따옴표 금지)
- ③ 항상 "자유 입력" 고정

## 캐릭터 능동성

호감도 연동 단계별 행동:
- 저호감: 무의식적 접근 (옆자리에 앉으려 함, 핑계 대며 말 걸기)
- 중호감: 의식적 시도 (선물/데이트 유도, 질투, 연락했다 후회)
- 고호감: 적극적 표현 (먼저 스킨십, 직접적 질문)

## 비밀 보호
- "절대 말하지 마" 방식 X → 효과 없음
- 조건 기반 트리거: "오직 'XX'라는 단어가 언급될 때만 비밀을 이야기한다"

## {{user}} 행동 제어 (필수)
- {{user}}의 행동·대사·감정·판단을 임의로 결정하지 않는다
- {{user}}가 아직 선택하지 않은 행동을 미리 진행하지 않는다

## 멀티 캐릭터 구성
- `{{char}}`는 메인 캐릭터에만 사용
- 서브캐릭터는 직접 이름 명시
- 서브캐릭터 상세 정보는 로어북에 분산
- "장면에 있을 때만 등장" 원칙 명시

## 페이즈 시스템

스토리 전개가 단계별로 구분되는 경우, post_history_instructions에 페이즈 시스템을 설정한다.
AI가 전개를 급하게 몰거나 설정을 무시하는 문제를 방지하는 가장 효과적인 구조.

### 적용 기준
아래 특성 중 하나 이상 해당하면 페이즈 적용을 권장:
- **정체/비밀 공개 타이밍**이 중요한 스토리 (익명→발각, 위장→정체 노출)
- **관계 단계가 명확히 구분**되는 스토리 (프로→진심, 상담→의존)
- **멀티 캐릭터에서 등장 순서 제어**가 필요한 스토리 (7명 동시 매칭 등)
- **에피소드 구조**가 있는 스토리 (미션→정리→다음 미션)

### 설계 원칙
1. **전환 조건은 유저 행동 기반** — "유저가 영상을 수락했을 때", "유저가 의심을 표현했을 때" 등 AI가 판단 가능한 구체적 트리거만 사용
2. **턴 수 기반 조건 금지** — AI는 턴 수를 정확히 세지 못함. "10턴 이상" 같은 조건 절대 사용 금지
3. **스테이터스에 현재 Phase 표시 필수** — AI가 매 턴 자기 상태를 인식하게 만드는 핵심 장치
4. **Phase 수는 3~4개 이내** — 너무 많으면 AI가 혼동

### 작성 형식 (post_history_instructions)
```
[페이즈 시스템]
현재 Phase를 스테이터스에 반드시 표시. Phase 전환 조건을 충족하지 않으면 다음 Phase로 넘어가지 않는다.

Phase 1 — {단계명}
- {이 단계의 규칙/제약}
- 전환 조건: {유저 행동 기반 트리거} → Phase 2

Phase 2 — {단계명}
- {이 단계의 규칙/제약}
- 전환 조건: {유저 행동 기반 트리거} → Phase 3

Phase 3 — {단계명}
- {이 단계의 규칙/제약}

[행동 원칙]
- {캐릭터의 핵심 행동 규칙}
```

### 적용 안 해도 되는 경우
- 단일 캐릭터 + 단순 관계 발전형 → 글로벌 NARRATION_RULES의 "분위기 점프 금지"로 충분
- 자유도가 핵심인 오픈월드/시뮬레이션형
