# Codex 검수 prompt 템플릿

도메인별 Codex 검수 prompt 템플릿. `codex:codex-rescue` 에이전트 호출 시 그대로 사용하거나 변형.

## 공통 규칙

모든 Codex prompt에 반드시 포함:

1. **검수 대상 파일 절대 경로**
2. **사전 컨텍스트** (작업의 목적, 사용자, 이전에 합의된 제약)
3. **검수 항목 6~10개** (도메인 특화)
4. **출력 형식 명시** (각 항목 1~3줄, 액션 아이템 "→ 액션:" 형식)
5. **신랄하게 가달라는 명시**: `"좋네요"는 빼고, 약점만 봅니다`

다단계 워크플로우에서는 **이전 Phase의 Codex 지적 + 반영 결과**도 prompt에 포함하여 동일 지적 반복 방지.

## 템플릿 1: 스토리/캐릭터 컨셉 검수

```
AChat 인터랙티브 픽션 스토리의 캐릭터 컨셉 문서를 검수해주세요.

## 컨텍스트

AChat은 Claude API 기반의 성인 인터랙티브 픽션 채팅 엔진입니다. 캐릭터 카드(description/personality/scenario/first_mes)와 로어북(키워드 트리거 + 상시 로어)을 기반으로 캐릭터가 사용자와 SSE 스트리밍으로 대화합니다.

지금 만들고 있는 스토리는 **{한 줄 컨셉}**입니다. 메인은 {메인 캐릭터}, 서브로 {서브 캐릭터}가 등장합니다. {시작 트리거}.

특이사항:
- {시스템 특이사항 1}
- {시스템 특이사항 2}

## 검수 요청 사항

`{파일 경로}` 를 읽고, 아래 관점에서 신랄하게 비판해주세요. "좋은데요"는 필요 없습니다. 약점만 보고 싶습니다.

1. **캐릭터 차별화** — N인이 실제로 다른 캐릭터로 읽힐 정도로 충분히 차별화됐는지. 비슷한 부분이 있다면 어떻게 더 갈라야 하는지.

2. **시나리오 흐름** — 페이즈가 자연스럽게 흘러갈 만한지. AI가 너무 빨리 점프할 위험은 없는지. 페이즈 트리거가 모호한 부분이 있는지.

3. **NSFW 톤** — 캐릭터별 욕망 톤 차별화가 실제로 작동할지. 묘사로 풀릴 때 진짜 다르게 느껴질지.

4. **이미지 시스템** — 상황 코드를 AI가 정확히 매핑해서 호출할 수 있을지. 헷갈리기 쉬운 코드에 대한 대응책 필요.

5. **외부 변수 활용** — {도메인별 외부 변수} 이게 실제로 작품 동력으로 쓰일 만한지. 단순 설정인지.

6. **위험 요소** — 컨셉 문서 마지막의 "위험 요소" 표에서 누락된 위험이 있는지.

7. **AI 채팅 실전성** — Claude {모델}이 이 컨셉을 받으면 실제로 캐릭터가 살아 움직일지, 아니면 천편일률적인 막친구물이 될지.

8. **삭제·축소 권고** — 이 컨셉 문서에서 빼야 할 부분, 너무 많이 적힌 부분이 있는지.

## 출력 형식

각 항목 1~3줄. 액션 아이템이 있으면 명시. 200~400단어 사이로 짧고 날카롭게.
```

## 템플릿 2: 스토리/프롬프트 검수 (2차)

```
AChat 인터랙티브 픽션 스토리의 프롬프트(02_prompt.md)를 검수해주세요. "좋네요"는 빼고 약점만 날카롭게 봐주세요.

## 컨텍스트

{도메인 컨텍스트 3~5줄}

스토리: {간략 요약}

특이사항:
- {시스템 특이사항}

## 사전 컨셉 검수

Codex가 이미 1차 컨셉에서 N개 약점 지적했고 모두 반영됨:
1) {지적 1 + 반영}
2) {지적 2 + 반영}
...

## 검수 요청 파일

`{파일 경로}`

원본 컨셉은 `{컨셉 파일 경로}`에 있습니다.

## 검수 항목 (각 1~3줄, 액션 아이템 명시)

1. **description 실전성** — description은 매 응답마다 시스템 프롬프트로 들어갑니다. {핵심 차별화 요소}이 실제 응답에 영향을 줄 정도로 명시적인지, 추상적 라벨에 그치는지.

2. **personality 적절성** — 길이·구체성·메인 캐릭터 중심성이 잘 잡혔는지.

3. **scenario 적절성** — 상황 설정의 명확성. AI가 첫 응답부터 길을 잃지 않을지.

4. **first_mes 보강** — 원본 보존 + 디테일 보강이 캐릭터를 더 살리는지 아니면 군더더기인지. 이미지 위치가 자연스러운지.

5. **post_history_instructions 강제력** — 매 응답마다 강제되는 지시사항이 너무 길어 AI가 흘려보낼 위험은 없는지. 줄일 곳이 있는지.

6. **상시 로어 토큰 효율** — constant: 1이라 매 응답마다 시스템 프롬프트에 박힘. 가장 토큰을 많이 잡는 항목, 통합·압축 가능한 항목.

7. **키워드 로어 트리거 정확성** — keys 선정 적절성. 트리거 누락·과잉 위험.

8. **시스템 호환성 위험** — `lib/context-builder.mjs`가 시스템 프롬프트를 조립할 때 충돌할 부분 (예: 자동 이미지 시스템 주입과의 충돌, hasImageMapping 정규식 매칭).

9. **누락 사항** — 컨셉 문서에는 있는데 프롬프트에는 빠진 것. 또는 프롬프트에 꼭 있어야 하는데 빠진 것.

10. **삭제 권고** — 빼야 할 부분, 과한 부분, 추상적이라 효과가 없을 부분.

## 출력 형식

각 항목 번호 매겨서 1~3줄. 액션 아이템 있으면 "→ 액션: ..." 형식으로 명시. 전체 400~600단어.

신랄하게 가주세요. "잘했다"는 필요 없고 약점만 봅니다.
```

## 템플릿 3: 시스템 설계 검수

```
{시스템 이름}의 설계 문서를 검수해주세요.

## 컨텍스트

{시스템 목적 / 사용자 / 운영 환경 3~5줄}

기술 스택: {스택}
규모: {DAU, QPS, 데이터 규모}
배포: {온프렘 / 클라우드 / 하이브리드}

## 검수 대상

`{파일 경로}`

## 사전 합의 사항

- {제약 1}
- {제약 2}
- {제약 3}

## 검수 항목

각 항목 2~5줄. 액션 아이템 명시.

1. **아키텍처 단순성** — 필요 이상으로 복잡한 부분이 있는가. YAGNI 위반.

2. **데이터 정합성** — 분산 트랜잭션·이벤트 순서·중복 처리 가정의 약점.

3. **장애 시나리오** — 단일 장애점, 복구 시간 목표(RTO), 부분 장애 시 동작.

4. **확장 한계** — 현재 설계가 N배 규모에서 무엇이 먼저 깨지는가.

5. **보안 경계** — 트러스트 바운더리 명시되었는가. 권한 분리 누락.

6. **운영 부담** — 당직·온콜·관측성 비용. 운영 자동화 가능성.

7. **마이그레이션 위험** — 기존 시스템에서 전환할 때 트래픽 분리·롤백·데이터 마이그레이션 시나리오.

8. **측정 가능성** — 시스템이 잘 동작하는지 어떻게 알 수 있는가. 핵심 지표 정의.

9. **숨겨진 가정** — "잘 될 것"이라는 낙관에 의존하는 부분.

10. **삭제 권고** — 빼야 할 부분 또는 후일 위임 가능한 부분.

## 출력 형식

각 항목 액션 아이템 명시 포함 2~5줄. 600~1000단어.

"좋다"는 빼고 약점만.
```

## 템플릿 4: 코드 리뷰 검수 (Codex가 다른 모델 작성 코드 리뷰)

```
다음 코드 변경을 검수해주세요.

## 컨텍스트

목적: {변경 목적}
영향 범위: {파일 / 모듈 / 시스템}
배경: {왜 이 변경이 필요한지}

## 검수 대상

- `{파일 1}` — {변경 사항 요약}
- `{파일 2}` — {변경 사항 요약}

## 사전 정보

- 기존 패턴: {코드베이스의 관행}
- 제약: {유지해야 할 것}
- 비목표: {이번에 안 하는 것}

## 검수 항목

각 항목 1~3줄, 라인 번호로 지목.

1. **정확성** — 로직 오류, 엣지 케이스 누락, 잘못된 가정.

2. **동시성** — race condition, deadlock, lock 누락.

3. **에러 처리** — 예외 처리 누락, 잘못된 fallback.

4. **보안** — 입력 검증, 권한 우회, 비밀 노출.

5. **성능** — N+1 쿼리, 메모리 누수, 불필요한 동기화.

6. **테스트 가능성** — 테스트 작성 가능한 구조인지. 모킹 필요 지점.

7. **가독성** — 네이밍, 추상화 수준, 주석 누락 / 과잉.

8. **API 호환성** — 외부 호출자에 미치는 영향.

9. **삭제 권고** — 데드 코드, 불필요한 추상화, 과한 방어 코드.

## 출력 형식

각 항목 라인 번호로 지목 + 짧은 설명 + 액션 ("→ 액션: ...").

신랄하게.
```

## 템플릿 5: 플랜/PRD 검수

```
다음 플랜/PRD를 검수해주세요.

## 컨텍스트

작업: {프로젝트 한 줄}
일정: {목표 일정}
팀: {담당자·규모}

## 검수 대상

`{파일 경로}`

## 검수 항목

각 항목 2~4줄, 액션 아이템 명시.

1. **사용자 가치 명확성** — "이게 왜 필요한가"가 페이지 첫 줄에 명확한가. 가치 가설이 검증 가능한가.

2. **가정의 약점** — "이렇게 될 것"이라는 가정 중 가장 흔들리기 쉬운 것.

3. **측정 가능성** — 성공 기준이 숫자로 정의되었는가. 측정 시기 명확한가.

4. **일정 현실성** — 의존성·블로커 고려된 일정인가. 버퍼 있는가.

5. **MVP 정의** — 핵심 가치를 검증할 수 있는 최소 단위로 정의되었는가.

6. **롤백 가능성** — 잘못됐을 때 되돌릴 수 있는가. 어디까지 되돌리는가.

7. **사각지대** — 명시 안 된 이해관계자, 누락된 사용 시나리오.

8. **삭제 권고** — 빼도 되는 범위, 후일 위임 가능한 부분.

## 출력 형식

각 항목 액션 명시. 400~700단어.

"좋다"는 빼고 약점만.
```

## 응용: Codex 응답 후 처리

Codex 응답 받은 뒤:

1. **반영 매트릭스 만들기**: 모든 지적을 표로 정리, 반영 위치 명시
2. **반영하지 않은 지적 명시**: 이유 포함
3. **수정 후 산출물 끝에 "Codex 협의 결과" 섹션** 추가

다음 Phase Codex 검수 시:
- 이전 Phase의 "Codex 협의 결과"를 그대로 prompt에 포함
- "사전 검수: 다음 N개 약점 지적됐고 모두 반영됨: ..." 형태로

이렇게 하면 Codex가 같은 지적 반복하지 않고 새로운 약점에 집중한다.
