# Persona × Codex — 다중 페르소나 합의 + Codex 신랄 검수 엔진

여러 전문가 페르소나가 각자의 관점에서 1차 산출물을 만들고, Codex(외부 AI)에게 신랄한 검수를 받아 보강한 뒤 사용자 게이트로 승인받는 **사이클 엔진**.

이 스킬은 **사이클의 유일한 정의처**다. `create-story`·`story-qa`는 이 엔진을 서브루틴으로 호출하기만 한다 — 사이클을 재서술하지 않는다. 사이클 규칙을 바꿔야 하면 이 파일만 고친다.

## 사용 시점

- 사용자가 "여러 페르소나로", "다양한 관점으로", "전문가 회의", "코덱스와 의견 나누면서", "깊이 있게"를 명시
- `create-story`·`story-qa`가 한 Phase의 산출물을 만들거나 검수할 때 (서브루틴 호출)
- 단일 시각으로 풀면 사각지대가 큰 복잡한 설계

사용하지 않는 경우: 단순 코드 수정·버그 픽스, 명확한 정답이 있는 lookup, 트리비얼 변경.

## 핵심 원칙

1. **페르소나는 동력 차이다** — 같은 대상을 봐도 서로 다른 결론을 내야 의미가 있다.
2. **Codex 검수는 신랄해야 한다** — "잘했다"는 필요 없다. 약점만.
3. **단계마다 사용자 게이트** — 보강본은 사용자 확인 후 다음으로.
4. **반영 추적 가능성** — Codex 지적은 1:1로 보강 내용과 매칭한다.
5. **중립 오케스트레이터** — 사이클을 돌리는 주체는 어떤 페르소나도 겸하지 않는다. 의견을 기계적으로 통합하고, 충돌은 해소하지 않고 사용자에게 올린다. 자기 초안을 방어하지 않는다.

## 실행 모드: 서브 에이전트 (페르소나 병렬 호출)

오케스트레이터(중립)가 페르소나를 `Agent` 도구로 **병렬 호출**한다 — 한 메시지에 여러 Agent 호출. 페르소나끼리는 통신하지 않는다. TeamCreate를 쓰지 않는다 (산출물이 단일이고, 배치 시 팀 재구성 비용이 과하다).

### 페르소나 호출 방법 (중요)

`.claude/agents/persona-*.md` 파일은 **세션 시작 시 로드**되어 `subagent_type`로 등록된다. 핫리로드되지 않으므로, 하네스를 방금 만든/수정한 세션에서는 `subagent_type: "persona-director"`가 "agent type not found"로 실패한다.

따라서 **정식 호출 방법은 `subagent_type: "general-purpose"` + 프롬프트 첫 줄에서 페르소나 정의 파일을 Read하도록 지시**하는 것이다. 이 방법은 세션 타이밍에 의존하지 않고 항상 동작한다.

```
Agent(
  subagent_type: "general-purpose", model: "opus",
  prompt: "먼저 `.claude/agents/persona-director.md`를 Read하고 그 페르소나 정의를 그대로 따르라.\n\n" + {mode, 대상, 도메인 컨텍스트, 지시}
)
```

세션을 재시작했고 에이전트가 등록됐다면 `subagent_type: "persona-*"`를 직접 써도 된다 — 동작은 같다. 확신이 없으면 general-purpose + Read를 쓴다.

## 페르소나 세트 (AChat 스토리 도메인)

`references/persona-catalog.md`의 도메인 1. 에이전트 파일은 `.claude/agents/persona-*.md`.

| ID | 페르소나 | subagent_type | 렌즈 |
|----|---------|---------------|------|
| D | 창작 디렉터 | `persona-director` | 톤·차별화·금기·후속확장 |
| A | AI 채팅 작가 | `persona-chat-writer` | 인터랙티브 구조·진입·분기·페이즈 |
| E | 엔진 QA 검수자 | `persona-engine-qa` | 엔진 호환·캐싱·로어북·토큰. 적대적 검수 전담 |
| N | NSFW 전문 작가 | `persona-nsfw-writer` | 성인 묘사·욕망 차별화·용어 로어 |
| P | 캐릭터 매력 설계자 | `persona-psychologist` | 욕망·태도·말투 선명함·행동 일관성·차별화 |
| K | 컬처 컨설턴트 | `persona-culture` | 도메인 디테일·호칭·말투·외부변수 |

다른 도메인(시스템 설계·코드 리뷰·플랜)은 catalog의 도메인 2~5 참조 — 그 경우 페르소나를 프롬프트 인라인으로 운용한다.

## 사이클 (1회)

호출자는 `{ mode: write | review, 대상, 도메인 컨텍스트 }`를 넘긴다.

### Phase 0 — 준비
- `mode`와 대상을 확인한다.
- `references/achat-engine.md`(엔진 제약)를 읽는다. 검수 모드면 `references/qa-checklist.md`도 읽는다.
- 비성인 스토리면 N 페르소나를 호출 목록에서 뺀다 (N은 어차피 "해당 없음"을 반환).

### Phase 1 — 페르소나 병렬 호출 → 1차 합의본
- 한 메시지에서 6개(또는 N 제외 5개) `Agent` 호출을 동시에 띄운다. 각 호출:
  - `subagent_type`: `general-purpose` / `model: "opus"` (호출 방법은 위 "페르소나 호출 방법" 참조)
  - prompt: 첫 줄에서 `.claude/agents/persona-*.md`를 Read하도록 지시 + `{ mode, 대상 경로 또는 스냅샷, 도메인 컨텍스트, 지시 }`
- 결과를 수집해 **기계적으로 통합**한다:
  - 작성 모드: 각 렌즈의 산출을 통합된 1차 산출물로 — 충돌하는 제안은 "충돌 사항"으로 표기, 해소하지 않는다.
  - 검수 모드: 각 렌즈의 약점을 `qa-checklist.md` 형식으로 병합 — 영역별 PASS/WARN/FAIL 표 + FAIL/WARN 목록. E의 FAIL은 다른 페르소나의 긍정 평가보다 우선한다.
- **1차 합의본 작성 후 즉시 Phase 2로** (사용자 게이트 없이).

### Phase 2 — Codex 신랄 검수
`codex:codex-rescue` 에이전트를 호출한다. prompt는 `references/codex-prompts.md`의 도메인 템플릿을 사용한다. 필수 포함:
1. 검수 대상 파일 절대 경로
2. 사전 컨텍스트 (목적·사용자·이전에 합의된 제약)
3. 검수 항목 6~10개 (도메인 특화)
4. 출력 형식 (각 항목 1~3줄, 액션 아이템 "→ 액션:")
5. "신랄하게, 약점만, 좋네요는 빼고" 명시

다단계 워크플로우면 **이전 Phase의 Codex 지적 + 반영 결과**를 prompt에 포함해 동일 지적 반복을 막는다.

### Phase 3 — 보강 + 반영 추적
Codex 지적을 1:1로 보강한다. 산출물 끝에 반영 매트릭스를 붙인다:
```
## Codex 협의 결과 (변경 요약)
| # | Codex 지적 | 반영 위치 / 방식 |
### 반영하지 않은 지적
- {지적} — 이유: {왜 반영 안 했는지}
```

### Phase 4 — 사용자 게이트
보강본을 제시한다. 형식:
```
## {Phase 이름} 완료 — {산출물 요약}
### 핵심 변경 (Codex 검수 반영)
### Codex 검수 반영 ({N}/{N})  — 반영 매트릭스
### 충돌 사항 (페르소나 간) — 있으면 선택 요청
전체: `{파일 경로}`
수정 사항 알려주시거나 "다음"이라고 하시면 진행합니다.
```
승인 시그널: "다음"·"ok"·"좋아"·"진행"·"승인". 거부/수정: 구체적 수정 요청.

## 예산 가드레일

사이클이 폭주하지 않도록:
- **페르소나 호출은 병렬 1회** — 같은 사이클에서 같은 페르소나를 재호출하지 않는다 (보강은 오케스트레이터가 한다).
- **Codex 호출은 사이클당 1회** — 재검수가 필요하면 그건 새 사이클이고 사용자 게이트를 거친다.
- **중단 기준** — 페르소나 과반(3+)이 실패하면 사이클을 멈추고 사용자에게 보고한다.
- **배치 처리** (story-qa의 11개 검수 등): 스토리당 사이클을 독립 반복한다. 한 스토리가 실패해도 다음으로 진행하고, 실패 스토리는 최종 요약에 명시한다.
- 토큰·시간 상한 수치는 첫 적용(파일럿)에서 측정해 정한다 — 추측으로 박지 않는다.

## 작성 모드 vs 검수 모드

| | 작성 모드 (write) | 검수 모드 (review) |
|--|------------------|-------------------|
| 페르소나 출력 | 렌즈별 산출물 | 렌즈별 약점 (qa-checklist 형식 강제) |
| 1차 합의본 | 통합 산출물 + 충돌 사항 | 영역별 PASS/WARN/FAIL + FAIL/WARN 목록 |
| Codex 템플릿 | codex-prompts 템플릿 1·2 (컨셉·프롬프트) | codex-prompts 템플릿 2 변형 (기존 스토리 약점) |
| 보강 | Codex 지적 반영해 산출물 개선 | Codex 지적 반영해 수정안 보강 |

## 다단계 워크플로우

복잡한 작업은 사이클을 여러 번 돈다 (예: 작성 = 컨셉 사이클 → 프롬프트 사이클 → QA 사이클). 각 사이클의 Codex 검수는 이전 사이클의 "Codex 협의 결과"를 컨텍스트로 전달한다.

## 레퍼런스

- `references/persona-catalog.md` — 페르소나 정의 (도메인 1 = AChat 6인)
- `references/codex-prompts.md` — 도메인별 Codex 검수 prompt 템플릿
- `references/achat-engine.md` — AChat 엔진 동작·제약 (페르소나 공용)
- `references/qa-checklist.md` — 검수 전용 객관 점검 기준 + 판정

## 에러 핸들링

| 상황 | 전략 |
|------|------|
| Codex가 칭찬 모드로 응답 | 재호출 + "신랄하게" 강조 (사이클당 1회 한도 내) |
| Codex 지적이 모순됨 | 그대로 사용자에게 보고, 사용자 결정 |
| 페르소나 의견이 강하게 충돌 | 1차 합의본에 "충돌 사항"으로 명시 + 게이트에서 선택 받기 |
| 페르소나 1~2명 실패 | 1회 재시도, 재실패 시 해당 렌즈 없이 진행 + 보고서에 누락 명시 |
| 페르소나 과반 실패 | 사이클 중단, 사용자에게 보고 |
| 사용자가 Codex 지적 반영 거부 | 거부 사유 기록 + "반영하지 않은 지적"에 명시 |
| 산출물이 너무 길어짐 | 보강 단계에서 Codex의 "축소 권고" 적극 반영 |
