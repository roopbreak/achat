## 현재 상태

사용자가 "큰 개편"으로 규정하고 **새 세션 검토를 명시 지시**. 직전 세션에서 설계 논의만 하고
실제 설계·구현은 새 세션으로 이연. 이 문서는 그 인계용.

**3분할 토대는 이미 배포됨** (상태창 본문 분리 작업 b6c0cf2~a2133e7). 빠진 것은:
① 스토리별 on/off·형식 설정, ② info부 상단화(중간 상태창 난제 근본 해결), ③ 선택지부 형식 설정.

**+ 커맨드부 추가 요청 (2026-06-11, 명확화됨)**: **`!디버그`/`!요약`/`!음란모드` 류 `!`-접두
시스템 명령어** — 서술 마커(`~행동~`)가 아니라 **앱이 인터셉트하는 기능 트리거**. 레지스트리
정의 → 어드민 등록 + 채팅 클릭 버튼 팔레트(즉시 실행). kind 3종: client_toggle(`!디버그`) /
server_action(`!요약`=summarizer) / mode_toggle(`!음란모드`=NSFW directive). 상세 plan §2-4.

**+ 분량/이어쓰기 메모 (2026-06-11)**: 글자수는 LLM 직접 제어 불가 — `max_tokens`(상한) +
`finishReason`(stop=자율종결 / length=잘림)뿐. 우리 "짧게 끝남"은 length 잘림이 아닌 모델
자율 stop이라 max_tokens 무관, 프롬프트 유도로만. 크랙식 **수동 "이어서 생성" 버튼**
(length 시 노출)을 auto-continue 자동발동 대신 검토 가치 있음(반복서술·대기 회피).

## 핵심 결정 근거 (보존)

- **info 상단화 = 게임 HUD 모델**: 상단 info는 "이전 턴의 결과". 사용자 수긍. 이어쓰기가
  본문 꼬리에서만 일어나 중간 상태 삽입이 구조적으로 불가 → P5 불필요화 가능.
- **선택지는 본문 끝 유지**: 읽기 순서 HUD→서술→선택. `splitChoices` 그대로 호환.
- **실험 우선**: 모델이 "상태→본문" 순서를 안정적으로 지키는지 미검증 → 코드 개편 전 프롬프트 실험.
