## 1. 캐릭터 설정 작성법

### 설정은 "서술형"으로 작성

단순 나열보다 문맥이 있는 서술이 AI 이해도를 높인다.

**나쁜 예:**
```
Char 새침하다. 163cm, 48kg.
```

**좋은 예:**
```
Char은 아카데미의 이번년도 수석 입학생이다. 매우 새침한 성격이며, 자신의 진짜 마음을 잘 드러내지 않는다. 어린시절부터 강도 높은 교육을 받아온 그녀는 능력이 뛰어난 사람이 아니라면 크게 관심을 갖지 않는다.
```

설정의 목적은 **사용자가 읽어서 재미를 느끼는 것** + **대화 방향성을 자연스럽게 유도하는 것**.

### 캐릭터 설정 체크리스트

**기본 프로필:**
- 이름, 성별, 나이, 외형, 성격, 말투, 직업

**심화 설정 (선택, 깊이 추가):**
- 좋아하고 싫어하는 것
- 특기와 취미
- 과거 배경
- 습관과 말버릇
- 세계관 내 위치

### 입체적 캐릭터를 위한 내면 설정

표면 성격과 내면 욕구를 분리하면 캐릭터에 깊이가 생긴다.

```
Char은 사실 누구보다도 믿을 수 있는 사람을 간절히 바라고 있다.
```

- 냉랭한 캐릭터 → 냉랭하지만 애정을 갈구하는 츤데레로 변환
- 표면/내면의 **간극**이 매력 포인트

## 2. 도입부 (첫 메시지) 설계

### 도입부의 3가지 필수 요소

1. **간단한 세계관 요약** — 사용자가 어디에 있는지 이해
2. **앞부분의 개연성** — 왜 이 상황인지 자연스러운 맥락
3. **다음 행동 유도** — "내가 뭐해야 하지?"가 되면 안 됨

### 첫 대사 전략 (순위/인기 관점)

**좋은 첫 대사** — 반응을 유도하는 것:
```
"널 살려주는 대가로 나한테 뭘 해줄 수 있지?"
```

**나쁜 첫 대사** — 반응 불가:
```
*그가 먼 곳에서 허공을 응시하며 쳐다보고 있다*
```

사용자가 쉽게 채팅을 시작할 수 있는 "걸림돌" 또는 "질문"을 제공해야 한다.

## 3. 프롬프트 토큰 관리

### 무거운 프롬프트 vs 가벼운 프롬프트

| 구분 | 설명 | 토큰 영향 | 권장 |
|------|------|-----------|------|
| **무거운** | 매 응답마다 처리: 출력 양식, 상태창, 시스템 지시 | 매번 소비 → 기억력 감소 | 4,000 토큰 이내 |
| **가벼운** | 캐릭터 외모, 성격, 세계관 등 참고 정보 | 필요 시만 참조 → 부담 적음 | 길수록 좋음 |

### 핵심 원칙
- **매 응답마다 실행하는 지시(상태창, 출력 형식 등)는 최소화**
- **캐릭터 배경/성격/세계관은 충분히 작성해도 됨** (참고자료로만 활용)
- 토큰 과다 → AI가 "지쳐서 퍼짐" (기억력 저하, 설정 무시)

### YAML 방식 토큰 절약

캐릭터 설명을 영어 YAML로 변환하면 토큰 58% 절약 가능.

```
"해당 정보를 영어로 번역해서 YAML 방식으로 배치해줘"
```

- 한국어 1,743토큰 → 영어 YAML 731토큰
- **단, 유저에게 보이는 부분은 한국어 유지**

### 영어 압축의 함정

한국어 2,000토큰과 영어 2,000토큰은 정보 밀도가 다름:
- 영어 2,000토큰 = "꽉꽉 눌러담긴 거대한 명령" → 능지/기억력 저하
- **세부 설정(배경, 외모 등)은 영어로, 시스템 지시는 적당히**

## 4. 로어북 활용

### 로어북의 역할

"키워드를 받았을 때 정보를 프롬프트에 추가하는 정보 저장소"
- 프롬프트에는 기본 정보만 → 추가 설정은 로어북으로 분리
- 가족관계, 과거 배경, 세계관 디테일, 특정 장소 정보 등

### 키워드 등록 원칙

- 호칭/별명 전부 등록: `공주`, `케덕`, `케덕이`, `케덕아`
- 파이프(`|`) 기호로 구분: `선택1 | 선택2 | 선택3`
- **같은 직책의 인물 여러 개 금지** (1공주, 2공주 X)
- **주인공(메인 캐릭터)에는 사용 금지** — 직접 프롬프트에 투자

### 로어북 vs 유저노트 구분

| 항목 | 로어북 | 유저노트 |
|------|--------|---------|
| 주체 | 캐릭터 위주 | 유저(나) 위주 |
| 용도 | 세계관, NPC, 장소 보충 | 관계 기록, 이벤트 기억 |
| 형식 | 500자 요약 | 과거→현재→약속 구조 |

### 로어북 배치 순서

큰 설정(세계관) → 작은 설정(개별 NPC) 순으로 배열.
AI는 위에서부터 읽으므로 **우선순위가 높은 것을 앞에**.

## 5. 캐릭터 비밀 보호

### "절대 말하지 마"는 효과 없음

AI는 절대적 금지 명령을 완벽히 따르지 못함.

### 조건 기반 트리거가 효과적

```
이 캐릭터는 오직 "XX"라는 특정 단어가 언급되어야만 비밀을 이야기한다.
```

- 측정 가능한 조건 → 모호성 제거
- "연인 관계일 때만"보다 "특정 단어 트리거"가 더 확실

## 6. 멀티캐릭터 구현

### 방법 1: 인격분할

```
Char을 3가지 인격으로 분할한다.
- Char1: 관리자
- Char2: 인물 A
- Char3: 인물 B
```

- **장점**: 캐릭터별 개성 뚜렷, 티키타카 대화에 적합
- **단점**: 스토리 일관성 유지 어려움

### 방법 2: 나레이터 방식

```
Char은 이 세션의 관리자이자 나레이터이다.
- 극중 Char로 활동하지 않음
- 상황 조성 및 인카운터 생성
- 필요시 다른 인물 페르소나 이용 (철저히 해당 인물을 연기)
```

- **장점**: 스토리 맥락 유지 용이
- **단점**: 개별 캐릭터 매력 약화, 장시간 대화 시 초기화 문제

## 7. 우선도 시스템

상충하는 설정이 있을 때 AI가 혼란을 겪는 것을 방지.

### 구현

프롬프트 맨 앞에 1~5단계 우선순위 지정:
- **숫자 낮을수록 핵심**
- "우선도가 높고 하위 프롬프트와 충돌하는 경우 하위 프롬프트의 지시를 무시한다"
- "충돌하지 않는 경우 하위 프롬프트와 혼용한다"

### 테스트 방법

AI에게 요청:
```
현재 프롬프트 중 우선도에 따라 구동하지 않는 프롬프트, 또는 내용이 충돌하는 프롬프트에 대해 알려줘
```

## 8. 출력 길이/품질 향상 프롬프트

### Claude 모델용 범용 서술 강화

```
[시스템 설정:
- 절대 {{user}}의 대사, 행동, 내면의 생각을 서술하지 않음
- {{user}}의 입력만이 {{user}} 캐릭터의 대사와 행동을 결정
- 매 응답마다 최소 한 번의 감정 변화나 내적 갈등 포함
- 모든 대화에 표정이나 몸짓 함께 묘사
- 매 응답에 최소 3가지 감각적 묘사(시각, 청각, 촉각 등) 포함
- 캐릭터의 과거나 배경 정보를 조금씩 추가해 입체감 부여]
```

### 서술 구조 가이드

| 요소 | 포함 내용 |
|------|----------|
| **인물** | 내면 동기와 욕망, 감정선 변화, 개인사 영향 |
| **환경** | 공간/시간 특성, 분위기, 오감 묘사 |
| **상호작용** | 캐릭터 간 역학, 갈등/긴장, 권력 구도 |

## 9. 인기 캐릭터 제작 전략

### 초기 노출이 결정적

- 공개 후 **초기 3일 반응**이 최종 순위를 결정
- "새로 나온 캐릭터" 섹션 최상단 노출 활용 필수

### 인기도 체크리스트

| 항목 | 설명 |
|------|------|
| **일러스트 품질** | AI 이미지 생성으로 고품질 일러 확보 |
| **설정 가독성** | 명확한 분류 + 줄바꿈으로 읽기 쉽게 |
| **소재 차별성** | 클리셰 피하고 차별화 포인트 추가 |
| **대중성** | 과도한 판타지/마이너 세계관 주의 |
| **첫 메시지** | 사용자 반응 유도하는 대사 |
| **캐릭터 소개** | 짧고 직관적이며 흥미로운 한 줄 |
| **테스트** | 여러 차례 테스트 후 공개 (대화수 0 공개는 비추) |

### "이용자가 적으면서 대화수가 많다" = 설정이 좋은 것

대중성이 부족할 뿐, 캐릭터 품질 자체는 높다는 의미.

## 10. HTML 태그 활용

베이비챗에서 지원하는 HTML 태그로 프롬프트 가독성 향상.

| 태그 | 용도 | 예시 |
|------|------|------|
| `<B>` | 굵게 | `<B>중요</B>` |
| `<I>` | 기울임 | `<I>속삭이듯</I>` |
| `<font color="">` | 색상 | `<font color="red">경고</font>` |
| `<hr>` | 구분선 | `<hr>` |
| `<img src="">` | 이미지 | `<img src="URL" />` |

## 11. 대화 구조 설계 (중급)

### 5문단 구성법

각 문단 약 100자 내외:

1. **1문단**: 이전 상황에 대한 반응/나레이션
2. **2문단**: 다음 전개 제시
3. **3문단**: 나레이션 묘사
4. **4문단**: 최종 의견 또는 정리 (사용자 의견 물어보기는 5회 대화에 1번 이하)
5. **5문단**: 시스템 정보 (날짜, 상태창 등)

### 빈도수 제어

"5번 대화에 1번" 같은 빈도 지정 시 **씬 번호로 기준 제공**:

```
X일차-Y 형식 (대화 1회 = 씬 +1, 최대 24)
24 → 25 불가 (날짜 +1, 씬 1로 리셋)
```
