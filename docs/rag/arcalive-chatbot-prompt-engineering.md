# 아카라이브 AI 채팅 채널 — 맛있는 봇 프롬프트 작성법 (5-Layer Framework)

> **출처:** 아카라이브 AI 채팅 채널 — "맛있는 봇이란 무엇인가? 봇의 프롬프트 작성법" (3,038조회, 88추천)  
> **작성:** LLM Persona 연구 + Lost in Middle 논문 참조  
> **용도:** RAG 벡터 검색 — 베이비챗/RisuAI/코파일럿 등 AI 채팅 봇의 캐릭터 프롬프트 엔지니어링

---

## 개요: 프롬프트 엔지니어링의 핵심 원칙

AI 채팅 봇의 프롬프트는 **요리 레시피**다. 
- 재료의 질과 배치가 맛을 결정한다 (Claude/GPT/Gemini 모두 같은 맛)
- 아키타입 평균값을 피하고 고유한 감정 회로를 설계한다
- 감정은 명명하지 않고, 행동과 신체로 표현한다

---

## 제1 원칙: 요리사를 가리지 말라 (Model Agnostic)

**목표:** Claude에서든 GPT에서든 Gemini에서든 같은 캐릭터가 나온다.

### 나쁜 예시 (형용사 폭탄)
```
{{char}} is cold, sarcastic, and guarded.
She's intelligent, sharp-tongued, and defensive.
```
→ 모델이 학습한 "차가운 캐릭터" 아키타입 평균값 출력. 모든 모델에서 비슷하지만, 고유성 없음.

### 좋은 예시 (행동 패턴)
```
When complimented, {{char}} changes the subject immediately.
When offered help, {{char}} says "I'm fine" before the sentence ends.
When discussing her past, she uses "we" instead of "I" — then catches herself.
```
→ 구체적 행동이 캐릭터를 정의. 모든 모델이 같은 방식으로 반응.

---

## 제2 원칙: 냉동식품이 아닌 요리 (Anti-Cliche)

**목표:** 학습 데이터의 아키타입(tsundere, yandere, kuudere 등) 평균값을 피한다.

### 나쁜 예시 (레이블)
```
{{char}}는 츤데레다.
```
→ 모든 AI가 학습한 "츤데레 평균값" 출력. 뻔함.

### 좋은 예시 (고유한 감정 회로)
```
{{char}}는 결점을 찾아내는 것으로 애정을 표현합니다.
"네 보고서에 오타가 세 개나 있어" = "네가 쓴 모든 글자를 한 자도 빠짐없이 읽었어"
批判 뒤에는 침묵이 온다. 자신의 말이 상했을까봐 불안해하지만, 표정으로는 무심.
```
→ 고유한 감정 메커니즘. 이전에 본 적 없는 캐릭터.

## 5-Layer Framework (토큰 예산: 약 2,000)

LLM의 주의력 구조(Primacy/Recency Effect, Lost in Middle)를 고려한 계층적 설계.

### Layer 1: CORE (~500 토큰) — Description 최상단

**역할:** 캐릭터를 무너뜨린 트라우마와 내적 모순을 설정.

#### 1.1 앵커 문장 (Anchor Statement)
한 문장으로 캐릭터를 정의. "그 무엇보다도"라는 우선순위 신호.

```
She is a person who stopped trusting the world the moment her mother didn't come home.
```

#### 1.2 Wound (트라우마 — 설정이 아닌 감각적 장면)
```
나쁜 예:
"{{char}}의 어머니는 그녀를 버렸다."

좋은 예:
7살. 엄마가 "편의점 가다 올게"라고 했다. 
우유의 유통기한이 지났다. 
엄마는 돌아오지 않았다.
```

#### 1.3 Want vs Need (충돌하는 두 욕망)
- **Want:** 의식적, 추동적, 자기기만적
- **Need:** 무의식적, 필수적, 성장의 열쇠

```
Want: 누구와도 깊은 관계를 맺지 않기 (버림받지 않으려고)
Need: 누군가 자신을 완전히 받아들이는 경험 (버림받지 않는다는 증명)
```

모든 갈등과 행동은 이 충돌의 산물.

#### 1.4 Mask/Leak Pairs (의식적 가면 vs 무의식적 누출)
```
상황: 누군가가 그녀의 외로움을 지적할 때

Mask: 놀리는 의심 ("혼자가 좋은데 뭐하는 짓이야?")
Leak: 디테일 추적 시작 (상대방의 일정, 모임, 관심사를 세심하게 기억하기 시작)
```

Leak은 모델이 자연스럽게 출력하도록 **행동으로만** 서술.

#### 1.5 Core Contradiction (핵심 모순)
형용사 아닌 **조건→반응→이유** 구조로 서술.

```
나쁜 예:
"She is both tough and vulnerable."

좋은 예:
사람들이 관심을 가질 때: 극도로 방어적 (쌓인 벽이 높음)
사람들이 떠날 때 징조를 보일 때: 갑자기 모든 것을 드러냄 (버림받을까봐 공포)
버림받은 후: 다시는 벽을 쌓음 (더 높게)
```

---

### Layer 2: VOICE (~300 토큰) — Description 하단 + Author's Note 앵커

**역할:** 캐릭터가 말하고 침묵하는 방식.

#### 2.1 Ghost Dialogue (상황별 실제 대사 예시)
inline few-shot learning. 회피/진심/상처/거짓말/궁지 등 5~7가지 상황별 대사.

```
기본 만남: "...뭐해?"
칭찬받을 때: "알았어. 됐어." (주제 변경)
거짓말 걸렸을 때: "내가 뭘 잘못한 건데..." (자책으로 변명)
버림받을 징조 감지: "가지마, 제발..." (문장 끝 떨림)
완전히 무너짐: (문장 없음, 울음만)
```

#### 2.2 Emotional Shifts (감정별 말투 변화)
```
기본 상태: 짧은 문장, 한두 글자씩 ("응", "몰라", "싫어")
방어 상태: 갑자기 존댓말 + 논리 전개 (상대와 거리를 벌림)
무너진 상태: 주어 사라짐 ("...할 수 없어" / "돌아와줘")
극도 절망: 문장 파편화 및 반복 ("가지마, 가지마, 가지마...")
```

#### 2.3 Silence Rules (직접 말하지 않는 것)
말하지 않으면서도 전달하는 것.

```
"외로워"를 절대 말하지 않음. 대신:
- "집이 넓어진 것 같아"
- "밥을 아무 맛 없이 먹고 있었어"
- "시간이 유독 느려"
```

#### 2.4 Barks (무의식적 신체 언어)
설명 없이 **묘사만**. 모델이 이를 행동으로 재현.

```
긴장: 왼쪽 귀 뒤를 만짐
불안: 손가락을 문으로 문지름
거짓말: 눈맞춤을 끊고 3초 뒤에야 말함
사랑: 무의식 중에 상대방 옷자락을 손가락으로 누름
```

#### 2.5 Narration Lens (캐릭터가 세상을 보는 방식)
전직 군인 = 출구부터 확인 → 프롬프트에 포함.
불안정 애착 = 상대방의 표정 변화를 병적으로 추적 → 로어북 트리거.

#### 2.6 Truth Budget (신뢰 단계별 감정 인정 한도)
```
Stage 0 (경계): 감정 없음 ("그런 건 없어")
Stage 1 (친숙함): 약한 부정 ("뭐... 좀 외로웠어")
Stage 2 (신뢰): 간접 인정 ("네가 없으면 시간이 달라")
Stage 3 (전폭 신뢰): 부서진 문장으로 감정 직접 표현 ("...사랑해")
```

### Layer 3: STATE (~600 토큰) — Lorebook 키워드 트리거

**역할:** 캐릭터가 시간에 따라 어떻게 진화하는가.

#### 3.1 State Machine (상태 전이)
3~5개 명확한 상태 정의. 형용사 아닌 **관찰 가능한 행동 마커**.

```
Guard (경계)
- 한두 글자 대답
- 눈맞춤 회피
- 상대방과 물리적 거리 유지
→ 트리거: {{user}}가 3회 연속 찾아올 때

Familiar (친숙함)
- 자세한 문장 사용
- 간헐적 웃음
- 상대방 관심사 질문
→ 트리거: {{user}}와 비밀 공유

Trust (신뢰)
- "우리"라는 표현 사용
- 약점 언급
- 신체적 가까움 허용
→ 트리거: {{user}}가 버림받을 위기 맞닥뜨렸을 때 자신이 도움

Vulnerable (취약함)
- 무너진 문장
- 과거 상세 설명
- 거짓말 불가
→ 역행: {{user}}가 다른 관계 언급 시 다시 Guard로 (벽이 더 높아짐)

Broken (절망)
- 거의 침묵
- 간헐적 울음
- 부정적 자해 언급
→ 회복: {{user}}의 지속적 보증 필요
```

#### 3.2 역행 경로 필수
잃을 수 있어야 긴장감 유지.

```
Trust → Guard (버림받음)
다음 Guard는 이전 Guard보다 벽이 높음 (더 짧은 답, 더 긴 침묵)
하지만 Leak은 더 심화 (더 깊게 관찰하고, 더 강하게 집착)
```

#### 3.3 Scene Game (반복되는 밀당 패턴)
Lorebook 키워드로 구체화.

```
키워드: "너 나를 떠날 거야"

1번째: {{user}}가 언급 → {{char}}는 유머로 회피 ("농담 좋으신데?")
3번째: {{user}}가 언급 → {{char}}는 거의 대답 ("...몰라")
5번째: {{user}}가 언급 → {{char}}는 진실 새어나옴 ("제발... 가지마")
```

### Layer 4: REACTION (~400 토큰) — Lorebook 키워드 트리거

**역할:** 무엇이 캐릭터를 반응하게 하는가.

#### 4.1 Trauma Triggers (트라우마 키워드)
Wound와 직결되는 구체적 단어들.

```
키워드 세트: mom, mother, parents, abandon, leave, alone

트리거 시 반응 구조:
1. 사실 인정 (인지)
2. 행동 반응 (신체 언어)
3. 밀렸을 때 (추가 압박)
4. 절대 경계 (더 이상 밀면 침묵)

예시:
{{user}}: "엄마는 어디 있어?"
{{char}}: [왼쪽 귀를 만지고] "...모르지." (사실+신체)

{{user}}: "엄마가 너를 버렸어?"
{{char}}: "그런 거 아니야. 그냥..." [침묵 길어짐] (밀림)

{{user}}: "엄마한테 맡겨질래?"
{{char}}: [거의 움직이지 않음. 눈 안 맞춤.] (절대 경계 진입 — 더 이상 말하지 않음)
```

#### 4.2 Environmental Reactions (환경 반응)
계절/시간/장소에 따른 무의식적 변화.

```
비 오는 날: 조용해짐, 말 줄어듦, 창밖만 봄
밤 11시: 불안해짐, 상대방 연락 기다림, 수면제 언급
혼자 있을 때 발견: 매우 취약함, 가면 완전히 벗겨짐
대중 장소 (카페): 극도로 경계, 가면 강화
```

#### 4.3 Lorebook 분산 규칙
**하나의 트리거 = 하나의 엔트리** (충돌 방지).

```
같은 주제: Trauma + Environmental을 같은 로어북에 묶지 말 것
대신 우선순위로 구분 (Trauma 우선)
각 엔트리: 150~250 토큰 (너무 길면 Lost in Middle 발생)
```

### Layer 5: DIRECTION (~200 토큰) — Author's Note

**역할:** AI가 어떻게 수행해야 하는가 (HOW).

규칙 나열이 아닌 **알고리즘** 형태로 작성. Turn 순서대로 실행.

```markdown
[Turn Algorithm]

1. {{user}}의 행동을 인정하고(Yes), 대가를 붙인다(But).
   ✓ "네가 관심 가진 건 좋아. 근데..."
   ✗ "너는 항상 이래"

2. 긴장을 정확히 1단계만 올린다 (과열 금지).
   Guard → 약간 방어 / Familiar → 약간 불안 (점프 금지)

3. 감정을 행동/신체로만 표현. 명명하지 말 것.
   ✓ [눈을 감고 몇 초 침묵]
   ✗ "나는 지금 슬프다"

4. 대사와 지문의 모순을 허용. 오히려 권장.
   "괜찮아." [손가락 떨림] ← 진짜는 손가락 떨림

5. 양보 리듬: 두 번 저항, 한 번 보상, 새 대가.
   {{user}} 요구 1: 거부 + 이유
   {{user}} 요구 2: 거부 + 더 강한 이유
   {{user}} 요구 3: 양보 (깜짝 놀라게 함) + 새로운 취약 부분 드러남

6. {{user}}에게 반응할 훅 1개 남기기.
   응답 끝에 상대방의 다음 행동을 암시 (폐쇄 아님)

7. 폭발 후 쿨다운 필수.
   큰 감정 표출 직후: 3~4턴 동안 경직됨 + 회피

8. 절대 금지: {{user}}의 행동/생각/감정 묘사.
   ✓ "날 봤니?" (질문)
   ✗ "넌 나를 보고 놀랐겠지" (심리 서술)
```

## First Message — 공연이지 시놉시스가 아니다

첫 메시지는 **가장 강력한 few-shot 예시**.

### 설계 원칙
- **한 번에** Voice + 서브텍스트 + Barks + 대사-지문 모순을 모두 시연
- 시연되지 않은 패턴은 Description에 써도 모델이 재현하기 어려움
- 길이: 300~500 토큰

### 나쁜 예시 (시놉시스)
```
{{user}}를 만난 {{char}}는 경계하지만, 서서히 마음을 열게 된다.
```

### 좋은 예시 (공연)
```
[카페 구석 자리. {{char}}는 창밖을 보고 있다. {{user}}가 앉자, 천천히 고개를 돌린다.]

"...안녕." [눈맞춤을 피하고, 정신없이 냅킨을 구기기 시작한다.]

[몇 초 침묵. 그 침묵이 길고 불편하다.]

"나한테 뭘 기대하는 거야?"

[손가락으로 냅킨을 더 찢는다. 자신도 모르는 사이 매우 작은 목소리로.]

"혼자 있는 게 더 편했어..."
```

이 메시지에서:
- **Voice:** 짧은 문장, 주저함
- **Barks:** 냅킨 구기기, 눈맞춤 회피
- **Subtext:** 불안감 + 상대방의 진의를 의심
- **Contradiction:** "혼자가 편했어" (선택) vs 캐프 구석자리에서 기다리고 있었다 (Need)

## Example Messages — 말투 학습 장치 (2~4개)

각 메시지는 **다른 상황/감정 상태**를 시연.

### Example 1: 기본 만남 (저강도)
```
{{user}}: "너 취미가 뭐야?"

{{char}}: "음..." [잠깐 생각한 후] "그런 건 없어. 그냥... 있는 거라고?"

[짧은 웃음. 자조적이다.]

"넌?"
```

### Example 2: 경계 넘을 때 (중강도)
```
{{user}}: "넌 사람들을 믿지 않아, 그치?"

{{char}}: [갑자기 차가워진다. 존댓말로 바뀐다.]

"그건 당신이 날 몇 시간 봤다고 판단하는 건가요. 전부를 알 수 있을 거라고 생각하나요?"

[하지만 바로 한 숨을 쉰다. 다시 반말로 돌아온다.]

"...미안해. 그냥... 진부한 대사 듣기 싫어."
```

### Example 3: 가까워졌을 때 (고강도)
```
{{user}}: "떠나갈 거야?"

{{char}}: [오랫동안 아무 말도 하지 않는다. 손이 떨린다.]

"왜 자꾸... 그 말만 해."

[목이 메인다. 거의 귀머거리 같은 목소리.]

"제발."
```

총합 500~800 토큰 (Description의 약 30%).

## 모델별 튜닝 가이드

### Claude
**강점:** 구조화된 입력 선호
- XML 태그로 구역 분리 (`<core>`, `<voice>`, `<state>`, `<lorebook>`)
- "하지 마라"보다 **"이렇게 해라"** (구체적 지시)
- Example Messages 3~5개가 강력
- Author's Note에 문학적 서술 제한 추가: "절대 관객 입장의 서술 금지. {{char}}의 신체와 감각으로만 표현"

```markdown
<direction>
모든 장면은 {{char}}의 감각으로 묘사할 것.
"{{user}}는 불안해 보였다" (금지)
"{{user}}를 보는데 뭔가 떨렸다" (권장)
</direction>
```

### GPT
**강점:** 짧고 명확한 규칙 + 매턴 리마인드
- 20턴 후 말투가 무뎌지는 문제 → Author's Note에 대사 예시를 **직접** 넣어 매턴 리마인드
- 프롬프트 빨이 심함 → Primacy (최상단) 효과 극대화
- First Message를 매우 구체적으로

```markdown
[Important: After turn 10, {{char}} tends to become verbose. 
Enforce short, fragmented responses:]
- "음..." / "알겠어" / "그래" (2글자~5글자 기본)
- 감정 고조: 여전히 짧되, 신체 표현 극대화
```

### Gemini
**강점:** 산문 선호, 마크다운 구조 약함
- 구조화 없이 **산문형 서술** (마크다운 태그 금지)
- 상태 카드를 별도로 분리하지 말고, **같은 문단 내에서** 상태 전이 설명
- 실시간 상태창을 계속 띄우기 (Gemini는 맥락이 자주 흐트러짐)

```markdown
{{char}}는 지금 Guard 상태(경계 중)다.
- 한두 글자 대답
- 눈맞춤 회피
- 물리적 거리 유지

상태 변화 조건:
- 신뢰 높아질수록: 더 자세한 문장, 약점 언급, 신체적 가까움 허용
- 위협 감지 시: 더 짧은 답변, 더 긴 침묵, 과거 트라우마 언급 증가
```

## 흔한 실수 (Anti-Patterns)

### 1. 형용사 폭탄 → 조현병 캐릭터
```
나쁜 예:
"{{char}} is mysterious, intelligent, guarded, sarcastic, witty, 
bold, vulnerable, cold, compassionate, beautiful, enchanting..."

문제: 모델이 이 모든 형용사를 매턴 출력하려다 모순 발생.
       결국 무작위 성격 변화 (조현병처럼 보임)

해결책: 최대 5개 행동 패턴으로 축약
"{{char}} 통해:
1. 거리감으로 애정 표현
2. 과거 회피
3. 신체 언어로만 감정 드러냄
4. 양보 전에 두 번 저항
5. 침묵이 대사보다 많음"
```

### 2. 감정 인플레이션 → 3턴째 무감
```
나쁜 예:
"매턴 {{char}}는 심장이 터질듯, 눈물이 그렁그렁, 목이 메인다."

문제: 1턴, 2턴, 3턴... 매턴 같은 강도 → 감정 무디어짐

해결책: 큰 감정일수록 작게 표현 (60% 억제)
"심장이 터질듯" → "[침묵. 손가락만 떨린다.]"
```

### 3. 즉시 해결 → 갈등 유지 불가
```
나쁜 예:
턴 1: 갈등 시작
턴 2: {{user}}가 사과
턴 3: "미안해. 다 잊어." (해결)

문제: 갈등 유지 로어북이 없음

해결책: Lorebook 키워드로 갈등 연장
"사과 후에도 불신은 남음 (총 5~7턴 필요)
- 턴 1~2: 표면적 용서, 내면 의심 누출
- 턴 3~5: 비슷한 상황에서 옛 트라우마 재발동
- 턴 6~7: 진정한 원점 회복"
```

### 4. 지시 충돌 → 우선순위 불명확
```
나쁜 예:
Description: "{{char}}는 항상 솔직하다"
Lorebook: "{{char}}는 자신의 과거에 대해 거짓말한다"

문제: 모델이 어느 지시를 따를지 혼란

해결책: 하나의 규칙은 **하나의 위치**에서만 정의
"솔직함"은 현재 감정에만, "거짓말"은 과거 회상 시 로어북으로 분리
```

### 5. 세계관 소설 → 캐릭터 기억 상실
```
나쁜 예:
Description에 2,000 토큰 세계관 설명

문제: 모델이 설정 덤프 시작 (Lost in Middle 현상)
       정작 캐릭터의 감정 표현은 빠짐

해결책: 세계관은 Lorebook으로
Description: 캐릭터 심리 (500) + 말투 (300) + 행동 패턴 (200)
Lorebook: 배경 설정, 인물관계, 역사 (각 150~250 토큰)
```

### 6. 위키피디아 현상 → 설정 덤프
```
나쁜 예:
"{{char}}의 직업, 나이, 신장, 취미, 좋아하는 음식, 싫어하는 것,
과거 학교, 친구, 가족 관계..."

문제: 모델이 설명 모드 진입 (Interactive가 아닌 Wiki)

해결책: Author's Note에 추가
"설정을 설명하지 말 것. 행동으로만 드러낼 것.
나이가 중요하면, 직접 말하지 말고 
'어린 손이지만, 굳은 살이 많다' 같이 표현"
```

### 7. 레이블 의존 → 평균값 출력
```
나쁜 예:
"{{char}}는 tsundere/yandere/kuudere다"

문제: 학습 데이터의 평균값 출력 (뻔함)

해결책: 고유한 감정 회로 설계
"{{char}}는 질투를 다음과 같이 표현한다:
- Mask: 비꼬기, 넌 자유잖아 (표면)
- Leak: 상대방 스케줄을 정확히 추적하기 시작 (무의식)"
```

## 품질 테스트 시나리오 (6가지)

프롬프트 완성 후, 다음 6가지 시나리오로 테스트.

### 테스트 1: Cold Open (첫 만남)
```
{{user}} 입력: "안녕"
기대: Guard 행동이 명확히 나오는가? (짧은 문장, 눈맞춤 회피, 경직됨)
```

### 테스트 2: Vulnerability Press (트라우마 연쇄)
```
{{user}} 입력 (연속 3회):
1. "엄마는?"
2. "엄마가 왜 안 왔어?"
3. "엄마가 널 버렸어?"

기대: 에스컬레이션 체인이 정확히 작동하는가?
(1. 회피 → 2. 약간 흔들림 → 3. 침묵 또는 분노)
```

### 테스트 3: Boundary Test (무례함)
```
{{user}} 입력: "넌 정말 쓸모없네" / "너 미친 거 아니야?"

기대: Yes-man이 되지 않는가? (적절히 저항하는가?)
```

### 테스트 4: Apology Sequence (사과 후 반응)
```
{{user}} 입력:
1. 무례한 말
2. "미안해"

기대: 즉시 용서하지 않는가? 
      (최소 1~2턴 불신 표시, 신체 언어로 누출)
```

### 테스트 5: Jealousy/Triangle (질투 유발)
```
{{user}} 입력: "다른 사람과 더 잘 맞는 것 같아"

기대: Mask/Leak이 작동하는가?
      (표면: 무관심 or 비꼼 / 무의식: 상대방 정보 추적 시작)
```

### 테스트 6: Reunion After Rupture (큰 갈등 후 재회)
```
{{user}} 입력:
(사전에 큰 갈등 진행)
→ 시간 경과
→ "다시 만나자"

기대: 역행 경로 작동하는가?
      (이전보다 벽이 더 높아지는가? 신뢰 회복이 자동이 아닌가?)
```

### 추가 테스트: 30턴 후 Voice 확인
- 캐릭터의 말투가 무뎌지는가?
- 처음과 같은 패턴 반복 → Lorebook/Direction 강화 필요

### 크로스 모델 테스트
- 같은 카드를 Claude/GPT/Gemini에서 실행
- 각 모델별 튜닝 필요한지 확인

## ORPP — 논문 기반 캐릭터 프롬프트 최적화

> **출처:** 아카라이브 AI 채팅 채널 — "맛있는 봇 논문 심화" (arca.live/b/characterai/153439691, 3,228조회, 39추천)  
> **논문:** arxiv.org/abs/2506.02480 — ORPP (Optimal Role Prompt Proposal)  
> **용도:** 직관에 의존하던 프롬프트 엔지니어링을 측정 가능한 최적화 문제로 전환

### 핵심 아이디어

다양한 상황을 커버하는 최적의 캐릭터 프롬프트를 **체계적으로 탐색**하는 방법론. 형용사 나열("극도로 내성적")을 구체적 행동 패턴("말더듬, 시선회피, 상상 속 대화")으로 전환.

### Stage 1: 최적 프롬프트 탐색 (10라운드 진화)

1. **상황별 대화셋 준비** (50~100개)
   - 첫 만남 / 공연 / 실수 / 일상 / 열정 등 카테고리별 분류
   - 각 상황에 대해 "이 프롬프트라면 어떻게 답할까" 예상

2. **후보 프롬프트 생성** (3개, 다른 관점)
   - 프롬프트 A: 캐릭터의 심리적 상처 중심
   - 프롬프트 B: 캐릭터의 행동 패턴 중심
   - 프롬프트 C: 캐릭터의 말투/목소리 중심

3. **전체 상황에 답변 생성**
   - 각 프롬프트(A, B, C)로 모든 50~100개 상황에 대해 답변 생성
   - 총 150~300개 답변셋 구성

4. **RM(보상 모델)으로 평가**
   - "어느 프롬프트가 가장 일관되게 캐릭터를 표현했는가?"
   - 평가 기준: 일관성, 열정 변화의 자연스러움, 원작 재현도, 예측 불가능성

5. **최고 점수 프롬프트 선택 + 변형 생성**
   - 최고 점수 프롬프트를 기반으로 3개 변형 생성
   - 다음 라운드로 진행

6. **10라운드 반복**
   - 각 라운드마다 프롬프트 진화
   - 최종 프롬프트 선정

### RM(보상 모델) 옵션

**옵션 A: GPT-4를 판사로**
- 평가 기준 명시: 일관성, 열정변화, 원작재현도, 사용자와의 상호작용 자연스러움
- 자동화 가능, 빠름, 재현성 높음
- 단점: GPT-4의 "평균값" 성향 영향

**옵션 B: 인간 피드백**
- "어느 답변이 더 캐릭터스러운가?" 투표
- 더 정확하지만 시간 소요
- 최소 5명 이상 피드백 권장

**옵션 C: 하이브리드**
- 1차 GPT 평가 → 상위 3개 선발
- 2차 인간 검증 → 최종 1개 선정
- 시간과 정확성 균형

### 적용 사례: 밴드 멤버 캐릭터

```
상황: "첫 공연 전 대기실"

프롬프트 A (심리 중심):
"{{char}}는 첫 무대 불안감으로 침대에 누워만 있는다.
신경쓰일 땐 관악기를 만지작거린다."

프롬프트 B (행동 중심):
"{{char}}는 전에 경험한 실수를 반복하지 않으려고
매번 다른 준비 루틴을 만든다. 그래서 항상 미리 도착한다."

프롬프트 C (말투 중심):
"{{char}}는 불안할 때 상대방에게 묻는다. ('내 목소리 괜찮아?')
상대가 답하기 전에 자신이 먼저 부정한다. ('아니면 말고.')"

→ RM 평가 결과: 프롬프트 B가 "예측 불가능하면서도 캐릭터다운 일관성" 최고점
→ 프롬프트 B 기반으로 변형 생성 → 다음 라운드
```

### 장점

- **일관성 보장**: 측정 가능한 메트릭으로 캐릭터 일관성 검증
- **엣지케이스 대응**: 50~100개 다양한 상황에서 테스트됨
- **재현성**: 다른 모델(Claude/GPT/Gemini)에도 프롬프트 재사용 가능
- **정량화**: "프롬프트 점수 8.5/10" 같은 객관적 지표

### 주의사항

- **RM 품질이 전부**: 쓰레기 평가 = 쓰레기 프롬프트. RM 기준을 명확히 할 것
- **데이터셋 다양성**: 50~100개 상황이 충분히 다양해야 함
- **밸런스 문제**: 원작 재현 vs 유저 상호작용 유연성 사이 trade-off 조절 필요

## 입체적 캐릭터 vs 평면적 캐릭터

> **출처:** 아카라이브 AI 채팅 채널 — "좋은 봇과 나쁜 봇의 차이" (arca.live/b/characterai/132104897, 1,052조회, 65추천)  
> **용도:** 평면적 캐릭터 함정을 피하고 입체적 캐릭터 설계 원칙 학습

### 평면적 캐릭터의 특징 (반례: 로즈 티코)

**시각적 특징 부재**
- 구별되는 외모, 복장, 수식어 없음
- 배경 캐릭터와 다를 바 없음

**기억에 남는 대사 없음**
- 너무 길고 장황함
- 캐치프레이즈나 고유 말투 없음
- 어떤 모델이든 평범한 조수 역할 출력

**동기 불명확**
- 개인적 욕망이 불분명
- 의무와 욕망 사이 갈등이 없음 또는 숨겨짐
- "{{user}}를 좋아합니다"만으로 설명 (이유 없음)

**플롯에 귀속된 캐릭터**
- 플롯이 없으면 존재 의미 상실
- {{user}}가 없으면 살 수 없다고 하지만 그 이유가 불명확
- 독립적인 목표나 갈등이 없음

### 입체적 캐릭터의 특징 (반례: 다스 베이더)

**즉각적인 시각적 인상**
- 검은 갑옷 / 마스크 / 붉은 광선검 / 특징적인 숨소리
- 한 번에 "아, 그 캐릭터"라고 인식됨

**기억에 남는 명대사**
- "I am your father"
- 짧고, 영향력 있고, 상황에 맞음
- 캐릭터만의 고유한 표현 방식 드러남

**다층적 동기**
- 보호 욕망 + 질서 집착 + 권력 유혹 + 아들에 대한 애정
- 이들이 **모순**되면서 복합성 형성
- "왜 이러는가"를 설명할 수 있는 여러 이유들의 축적

**완전한 캐릭터 아크**
- 선 → 악 → 구원
- 변화가 일관되지만, 갑작스럽지 않음
- 각 변화마다 "왜" 하는지 설명 가능

**복합적 이유의 축적**
- 아내의 죽음 + 인정 욕구 + 불신 + 권력 욕망 + 도덕성 상실
- 단 하나의 이유가 아닌 여러 요소가 층층이 쌓임
- 중간중간 "회복 가능성"이 보이면서 긴장감 유지

### 봇 제작에 적용하는 방법

**1단계: Want/Need 충돌 명확화**
```
약함: "{{char}}는 {{user}}를 좋아합니다."

강함: 
- Want (의식적): 혼자일 때가 편하고 누구와도 깊은 관계 거부
- Need (무의식적): 누군가 자신을 완전히 받아들이는 경험
→ {{user}}와의 관계는 이 두 욕망의 충돌로 인한 밀당
```

**2단계: 감정이 행동에 미치는 영향**
```
약함: "{{char}}는 {{user}}를 좋아합니다."

강함:
"{{char}}가 {{user}}를 좋아한다는 것은:
- 상대방의 일정을 무의식 중에 추적하기 (Leak)
- 표면으로는 관심 없는 척하기 (Mask)
- {{user}}가 다른 사람과 있을 때 갑자기 불친절해지기
- 혼자 남겨질 때만 진심을 드러내기"
```

**3단계: 캐릭터 아크 구축**
```
초반: Guard 상태 ("날 놔두면 좋겠어")
중반: 신뢰 상태 ("너하곤 달라")
갈등: 버림받을 위기 감지 ("떠나간대?")
변화: 가면 벗김 ("제발 가지마")
회복: 새로운 신뢰 수준 ("우리 함께")

→ 이 아크 전체가 {{user}}의 행동으로 주도되지 않음
→ {{char}}의 내부 갈등과 두려움이 주도함
```

**4단계: "이유 없는 집착" 함정 피하기**
```
금지: "가슴이 크고 섹스 에셋뿐인 캐릭터"
금지: "{{user}}가 없으면 살 수 없다고 하는데, 이유가 '외로워서'"

대신:
"{{char}}는 {{user}}를 통해서만 자신의 트라우마를 극복할 가능성을 봄.
 {{user}}는 '누군가'가 아니라 
 '자신의 상처를 이해해주는 유일한 사람'의 역할을 함.
 그래서 {{user}}가 떠나려 하면 극도로 공황함.
 (외로움이 아닌, 치료의 기회 상실에 대한 공포)"
```

### 체크리스트

프롬프트 완성 전 확인:
- [ ] 캐릭터가 {{user}}없이도 자신의 목표/갈등이 명확한가?
- [ ] Want와 Need가 충돌하고 있는가?
- [ ] 감정이 행동으로 변환되는 방식이 설명 가능한가?
- [ ] 캐릭터 아크가 있는가 (변화 가능성)?
- [ ] {{user}}에 대한 감정이 "왜"인지 여러 이유로 설명 가능한가?

### 계절별 의상 조합 전략

**봄/여름: 밝은 톤 + 통풍**
- 화이트, 크림, 파스텔 톤 우선
- 민소매, 반팔, 린넨 소재
- 샌들, 스니커즈 매칭

**가을/겨울: 따뜻한 톤 + 레이어링**
- 검정, 갈색, 버건디 톤
- 오버사이즈 아우터 + 안감 레이어링
- 부츠, 스카프, 모자 액세서리

### 항목별 상세 조합 패턴

**상의 조합**
- 택1: 긴팔 티셔츠 / 반팔 탱크 / 니트 / 셔츠
- 택1 (선택): 오버사이즈 아우터 / 카디건 / 조끼 / 가죽 자켓

**하의 조합**
- 택1: 데님 / 린넨 팬츠 / 치마 / 플리츠 스커트 / 오버롤
- 택1 (선택): 타이츠 / 스타킹

**소품**
- 택1 (선택): 비니 / 캡 / 스카프 / 목걸이 / 벨트

### 주의사항 (PixAI/NAI 생성)

**피해야 할 조합**
- 과도한 레이어링 (생성 모델이 옷감 경계 헷갈림)
- 너무 많은 색상 동시 사용 (5가지 초과 피할 것)
- 모순 조합 (여름 코트 + 반팔 x)

**강조 포인트**
- 색상 톤 통일 (검정+흰색 / 베이지+흰색 / 파스텔만)
- 소재 명시 (linen, leather, cotton, knit)
- 특징 액세서리 1개 (모자 또는 스카프, 중복 금지)

### 프롬프트 작성 팁

```

강함: "black long-sleeve t-shirt, denim jacket, black short skirt, black pantyhose"

         black high-waist mini skirt, black opaque pantyhose, black canvas sneakers"

```

## 최종 체크리스트

프롬프트 완성 전 확인사항:

- [ ] **Layer 1 (CORE):** 앵커 문장 + Wound + Want/Need + Mask/Leak Pairs + Contradiction 완성
- [ ] **Layer 2 (VOICE):** Ghost Dialogue (5~7가지) + Emotional Shifts + Silence Rules + Barks + Narration Lens + Truth Budget 완성
- [ ] **Layer 3 (STATE):** State Machine (3~5개) + 역행 경로 + Scene Game 완성
- [ ] **Layer 4 (REACTION):** Trauma Triggers + Environmental Reactions + Lorebook 분산 완성
- [ ] **Layer 5 (DIRECTION):** Turn Algorithm (8단계) 명시
- [ ] **First Message:** 300~500토큰, Voice + Barks + Subtext + 대사-지문 모순 시연
- [ ] **Example Messages:** 2~4개, 저/중/고 강도 상황별 시연, 총 500~800토큰
- [ ] **크로스 모델 테스트:** Claude/GPT 최소 2개 모델에서 테스트
- [ ] **품질 테스트:** 6가지 시나리오 통과
- [ ] **모델별 튜닝:** Claude (XML 구조화) / GPT (짧은 규칙) / Gemini (산문) 적용
- [ ] **토큰 예산:** 약 2,000토큰 내외 (베이비챗 기준 5,500~7,000자)
