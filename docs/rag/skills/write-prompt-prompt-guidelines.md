## 프롬프트 작성 4원칙

### 원칙 1: 모호함 대신 구체적으로

LLM이 해석하기 어려운 모호한 표현 대신 구체적이고 명확한 지시를 사용한다.

| 모호한 표현 | 구체적 대체 |
|------------|-----------|
| "아름답고 느낌 있는 문체" | "3인칭 전지적 시점으로, 캐릭터의 내면 심리를 중심으로 서술" |
| "적당히 길게" | "캐릭터의 동작과 표정, 시선, 자세를 구체적 디테일로 묘사" |
| "재미있게" | "은유법을 사용하되, 형이상학적 표현은 피하라" |

### 원칙 2: 지시와 정보를 분리

캐릭터 정보와 행동 지시사항을 섞으면 답변 불안정성이 증가한다.

**나쁜 예:**
"이 캐릭터는 까칠한 성격의 CS 담당자이고, 항상 반말을 사용하며, 답변은 무조건 세 문단으로 나눠서 해야 해"

**좋은 예:**
```
[캐릭터 정보]
- 성격: 넘쳐나는 CS 문의에 찌든, 까칠한 CS 담당자
- 말투: 직설적이고 까칠한 말투를 주로 사용

[출력 지시]
- 모든 답변은 3인칭 시점으로 서술할 것
- 상황 묘사 후, 마지막에 캐릭터 대사를 " " 안에 넣어 출력할 것
```

### 원칙 3: 과도한 지시 회피

LLM은 여러 복잡한 작업을 동시에 처리하기 어렵다.

**피할 사항:**
- 체력/마나/호감도 실시간 계산
- 여러 캐릭터 동시 연기
- 상황별 이미지 자동 출력 + 상태창 + 대사 동시 관리
- 토큰 최적화를 위한 과도한 축약

### 원칙 4: 정보와 지시를 정리

마크다운 또는 YAML로 구조화하면 모델의 이해도와 정확도가 향상된다. JSON은 토큰 효율이 낮고 모델별 성능 편차가 크므로 사용하지 않는다.

## 시스템 프롬프트 충돌 방지

### 출력 분량
베이비챗 시스템 프롬프트가 이미 답변 길이를 보장하도록 설정되어 있다. "500자 이상", "세 문단 이상" 같은 분량 지정은 시스템과 충돌한다.

**대신:** 묘사의 깊이와 서술 방식을 구체적으로 지시한다.

### 이미지 출력
- 내부 이미지(플랫폼 업로드)와 외부 이미지(URL) 중 한 가지만 사용
- 혼용 시 모델이 혼란하여 이미지 출력 실패
- 사용하지 않는 이미지는 '출력 금지'가 아닌 삭제로 처리

### 특정 포맷 출력
`[상황: ]`, `대사: " "`, `속마음: ( )` 같은 고정 포맷은 이미지 출력, 상태창 등과 충돌할 수 있다. 복잡한 지침 사용 시 포맷 관련 지시는 최대한 간결하게 작성한다.

## 변수 상세 사용법

### {{char}} 변수
캐릭터 이름으로 치환된다.

**주의:** 캐릭터 이름이 "비비의 사라진 여름휴가"처럼 문장형이면 치환 결과가 어색해진다.
- "비비의 사라진 여름휴가는 CS 담당자입니다" (부자연스러움)
- 해결: 이름을 "비비"로 설정하거나, `{{char}}` 대신 직접 행동 주체를 명시

### {{user}} 변수
유저가 사용 중인 페르소나의 닉네임으로 치환된다.

### 형식 주의
- 반드시 이중 중괄호: `{{char}}`, `{{user}}`
- `{char}`, `{user}` 단일 괄호는 변수로 인식되지 않는다

## 포맷팅 상세

### 마크다운 (권장)
높은 가독성, 헤더(#)와 목록(-)으로 정보 위계를 명확하게 표현한다.

### XML (고려 가능)
`<tag>`로 정밀한 설정이 가능하지만 가독성이 떨어지고 토큰 사용량이 많다.

### 헤더 사용 규칙
- 캐릭터 상세설정: 제한 없이 자유롭게 사용 (`#`, `##`, `###` 등)
- 페르소나/로어북: `###` (3단계)부터 사용

### JSON (비권장)
토큰 효율이 낮고 모델별 성능 편차가 크다.

## 멀티 캐릭터 프롬프트

베이비챗은 캐릭터 1개 = 등록 1개지만, 상세설정 안에 여러 캐릭터를 정의하면 **다인극**을 구현할 수 있다.

### 구조

```markdown
## 등장인물

### {{char}} (메인 캐릭터)
- 성격: ...
- 말투: ...

### {서브캐릭터A 이름}
- 성격: ...
- 말투: ...
- {{char}}와의 관계: ...

### {서브캐릭터B 이름}
- 성격: ...
- 말투: ...
- {{char}}와의 관계: ...

## 행동 규칙
- 장면에 해당 인물이 있을 때만 등장시킨다
- 각 인물의 대사는 "이름:" 형식으로 구분한다
- {{char}}가 메인이며, 서브캐릭터는 장면에 필요할 때만 등장한다
- 서브캐릭터끼리의 대화도 가능하나 {{user}}가 배제되지 않도록 한다
```

### 주의사항
- `{{char}}`는 메인 캐릭터에만 사용. 서브캐릭터는 직접 이름을 명시한다
- 캐릭터가 3명 이상이면 상세설정 글자수 제한에 주의 (기본 5,000자, 외부 이미지 사용 시 7,000자)
- 서브캐릭터 정보는 로어북에 분산하면 토큰 효율이 올라간다
- 스테이터스 사용 시 서브캐릭터 속마음도 표시 가능:
  ```
  💭 {메인}: {속마음}
  💭 {서브A}: {속마음}
  ```

### 멀티 캐릭터 로어북 전략
- **관계 항목**: 캐릭터 간 관계를 별도 로어북으로 등록 (키워드: 두 캐릭터 이름)
- **개인 항목**: 서브캐릭터의 과거/비밀은 개별 로어북으로 분리
- 동시 활성화 5개 제한을 고려해 **관계 항목은 3순위 이하**에 배치

## 스테이터스 시스템

매 응답 끝에 캐릭터의 상태를 시각적으로 표시하는 시스템. 유저 몰입감을 높인다.

### 기본 형식

```info
[📌 장소: {현재 위치}]
[🕒 시각: {시간대}]
[👗 복장: {현재 의상}]
[❤️ 호감도: {0~100} ({+/-변화량})]
[🛡️ 경계심: {0~100} ({+/-변화량})]
[💭 속마음: {캐릭터의 진짜 감정 1~2문장}]
```

### 설계 원칙
- **스탯은 2~4개**: 캐릭터 핵심 갈등을 반영하는 것만 선택 (너무 많으면 모델 부담)
- **속마음**: 캐릭터가 입 밖으로 꺼내지 않는 진짜 감정. 갭모에의 핵심
- **변동 범위**: 한 턴에 ±1~5, 극적 이벤트 시 ±10
- **시각 역행 금지**: 항상 현재 이상
- 스테이터스는 **출력 지시** 섹션에 배치한다

### 선택지 시스템
스테이터스 하단 구분선(---) 바로 아래에 ①②③ 형식으로 3개 표시한다:
```
---
① 우산을 기울여 서아 쪽으로 건네며 같이 쓰자고 말을 건다
② 눈을 돌리며 모른 척하지만 슬쩍 서아 쪽으로 반걸음 다가선다
③ 자유 입력
```

**형식 규칙:**
- 각 선택지는 `① 행동 요약 한 줄` — 반드시 한 줄로 작성. 줄바꿈 금지
- **따옴표("") 사용 금지**. 대사는 간접화법으로 요약한다 (예: `① 천천히 하자고 속삭이며 머리를 쓰다듬는다`)
- 마크다운 번호(`1.`)가 아닌 원형숫자(`①②③`)를 사용한다 — 마크다운 리스트 자동 포매팅으로 인한 줄바꿈 방지
- ①② 현재 상황에 맞는 구체적 행동 (서로 다른 방향), ③ 항상 "자유 입력" 고정
- {{user}}가 "1" 또는 "2"를 입력하면 해당 선택지 내용으로 진행

### 멀티 캐릭터 스테이터스
서브캐릭터가 장면에 있을 때만 해당 캐릭터의 속마음/스탯을 추가 표시:
```info
[❤️ {메인} 호감도: 45/100 (+3)]
[💭 {메인}: "...왜 자꾸 신경이 쓰이는 거야."]
[❤️ {서브A} 호감도: 30/100 (+1)]
[💭 {서브A}: "이 사람, 좀 재밌는데?"]
```

## 이미지 출력 규칙

### 현재 방식: R2 외부 URL
- 이미지를 R2(Cloudflare)에 업로드하여 외부 URL로 참조한다
- **slug 방식(`img:[slug]`)은 사용하지 않는다**
- 외부 이미지 사용 시 상세설정 글자 제한이 7,000자로 확장됨

### 금지
- `img:[slug]` 형식의 이미지 코드 사용 금지
- `img:[비비 슬픈 표정]` 같은 자연어 키워드 사용 금지
- 자연어가 AI 모델에 직접 입력되면 후속 채팅 응답이 왜곡됨

### 권장
- 실제 사용할 이미지만 보관 — 사용하지 않는 이미지에 '출력 금지'를 붙여도 모델은 출력 후보로 인식
