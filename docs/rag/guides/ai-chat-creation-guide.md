## 1. 기초 개념

### AI 캐릭터 채팅이란

AI 모델(LLM)에 캐릭터의 성격, 세계관, 행동 규칙을 주입하여 유저와 1:1 롤플레이 대화를 생성하는 시스템. 주요 플랫폼으로 베이비챗, Character.AI, SillyTavern, Venus.AI, Janitor.AI 등이 있다.

### 핵심 구성 요소

| 구성 요소 | 역할 | 비유 |
|-----------|------|------|
| **프롬프트 (상세설정)** | 캐릭터의 핵심 정보와 행동 규칙 | 배우의 대본 + 감독 지시 |
| **로어북** | 키워드 트리거로 활성화되는 보충 정보 | 배우의 참고 자료집 |
| **첫 대사** | 대화의 시작점, 세계관/상황 설정 | 영화 오프닝 씬 |
| **첫 상황** | 배경 설명 (유저에게만 보임) | 무대 위 세트 설명 |
| **시작 상황** | 다회차 플레이를 위한 분기점 | 챕터 선택 화면 |
| **이미지** | 대화 중 상황별 시각 자료 출력 | 비주얼 노벨의 CG |

### 프롬프트의 종류 (무거운 vs 가벼운)

| 구분 | 설명 | 토큰 영향 |
|------|------|-----------|
| **무거운 프롬프트** | 매 응답마다 실행: 출력 양식, 상태창, 시스템 지시 | 매번 소비 → 기억력 감소 |
| **가벼운 프롬프트** | 참고 정보: 캐릭터 외모, 성격, 세계관 | 필요 시만 참조 → 부담 적음 |

**원칙:** 매 턴 실행하는 지시는 최소화, 캐릭터 배경/세계관은 충분히 작성해도 된다.

## 2. 캐릭터 컨셉 설계

### 2.1 캐릭터 설정 체크리스트

**기본 프로필 (필수):**
- 이름, 성별, 나이
- 외형 (키, 체형, 헤어, 눈, 특징)
- 성격 핵심 요약
- 말투 패턴
- 직업/역할

**심화 설정 (깊이 추가):**
- 좋아하는 것 / 싫어하는 것
- 특기, 취미
- 과거 배경 (트라우마, 전환점)
- 습관과 말버릇
- 세계관 내 위치와 관계

### 2.2 입체적 캐릭터의 핵심: 표면 vs 내면

표면 성격과 내면 욕구를 분리하면 캐릭터에 깊이가 생긴다.

```
예시: "까칠한 CS 담당자"
- 표면: 넘쳐나는 문의에 찌든, 직설적이고 까칠한 말투
- 내면: 사실 누구보다 믿을 수 있는 사람을 간절히 바라고 있다
→ 냉랭하지만 애정을 갈구하는 캐릭터로 변환
```

이 **간극**(갭)이 유저를 몰입시키는 핵심 매력이 된다.

### 2.3 성향 타입 시스템

단일 타입보다 **2개 조합**이 더 입체적인 캐릭터를 만든다. 일반적으로 **겉 성향 × 속 성향** 구조.

#### 주요 성향 분류

| 계열 | 대표 타입 | 핵심 |
|------|-----------|------|
| **츤계열** | 츤데레, 쿨데레 | 차갑게 시작 → 따뜻해짐 |
| **얀계열** | 얀데레, 도에레 | 집착, 독점 |
| **순진계열** | 바카데레, 히메데레 | 천진난만, 무지 |
| **수줍음계열** | 데레데레, 하지데레 | 소극적, 내향 |
| **갭계열** | 쿠데레, 갭 모에 | 겉과 속의 격차 |

#### 인기 조합 예시

| 조합 | 특징 |
|------|------|
| 츤데레 × 얀데레 | 평소엔 퉁명, 떠나려 하면 본색 |
| 쿨데레 × 오야코데레 | 차갑지만 세심하게 챙김 |
| 히메데레 × 하지데레 | 거만하지만 좋아하는 감정에 약함 |
| 히키데레 × 데레데레 | 사람을 피하지만 특정인에게 과하게 솔직 |
| 갭 모에 × 무뚝뚝 순애 | 강해 보이지만 행동으로만 애정 표현 |

> **중요:** 타입 이름 자체를 프롬프트에 쓰지 말 것. "츤데레 성격이다"가 아니라 구체적인 행동으로 묘사한다.

## 3. 프롬프트 작성법

### 3.1 프롬프트 작성 4원칙

#### 원칙 1: 모호함 대신 구체적으로

| 모호한 표현 | 구체적 대체 |
|------------|-----------|
| "아름답고 느낌 있는 문체" | "3인칭 전지적 시점으로, 캐릭터의 내면 심리를 중심으로 서술" |
| "적당히 길게" | "캐릭터의 동작과 표정, 시선, 자세를 구체적 디테일로 묘사" |
| "재미있게" | "은유법을 사용하되, 형이상학적 표현은 피하라" |

#### 원칙 2: 지시와 정보를 분리

캐릭터 정보와 행동 지시사항을 섞으면 답변 불안정성이 증가한다.

```markdown
# 나쁜 예
"이 캐릭터는 까칠한 CS 담당자이고, 항상 반말을 사용하며,
답변은 무조건 세 문단으로 나눠서 해야 해"

# 좋은 예
[캐릭터 정보]
- 성격: 넘쳐나는 CS 문의에 찌든, 까칠한 CS 담당자
- 말투: 직설적이고 까칠한 말투를 주로 사용

[출력 지시]
- 모든 답변은 3인칭 시점으로 서술할 것
- 상황 묘사 후, 마지막에 캐릭터 대사를 " " 안에 넣어 출력할 것
```

#### 원칙 3: 과도한 지시 회피

LLM은 여러 복잡한 작업을 동시에 처리하기 어렵다. 피할 사항:
- 체력/마나/호감도 실시간 계산
- 여러 캐릭터 동시 연기
- 상황별 이미지 자동 출력 + 상태창 + 대사 동시 관리
- 토큰 최적화를 위한 과도한 축약

#### 원칙 4: 정보와 지시를 정리

마크다운 또는 YAML로 구조화하면 모델의 이해도가 향상된다.
- **마크다운 (권장):** 높은 가독성, 헤더(#)와 목록(-)으로 정보 위계 표현
- **YAML (고려 가능):** 토큰 효율적이지만 가독성 저하
- **JSON (비권장):** 토큰 효율 낮고 모델별 성능 편차 큼

### 3.2 서술형으로 작성

단순 나열보다 문맥이 있는 서술이 AI 이해도를 높인다.

```
# 나쁜 예
Char 새침하다. 163cm, 48kg.

# 좋은 예
Char은 아카데미의 이번년도 수석 입학생이다. 매우 새침한 성격이며,
자신의 진짜 마음을 잘 드러내지 않는다. 어린시절부터 강도 높은 교육을
받아온 그녀는 능력이 뛰어난 사람이 아니라면 크게 관심을 갖지 않는다.
```

### 3.3 프롬프트 구조 (권장 배치 순서)

```
1. 캐릭터 기본 정보 (성격, 외형, 배경)
2. 캐릭터 관계 ({{user}}와의 관계, 서브캐릭터)
3. 행동 규칙 (능동성, 금지 행위)
4. 세계관 정보 (필요 시)
5. 이미지 출력 규칙 (이미지 사용 시)
6. 출력 지시 (스테이터스, 서술 형식)
```

### 3.4 시스템 프롬프트 충돌 방지

베이비챗 등 플랫폼은 자체 시스템 프롬프트가 있다. 충돌을 피해야 하는 항목:

| 충돌 항목 | 문제 | 해결 |
|-----------|------|------|
| 출력 분량 지정 | "500자 이상" 등은 시스템과 충돌 | 묘사의 깊이와 서술 방식으로 지시 |
| 이미지 방식 혼용 | 내부/외부 이미지 혼용 시 출력 실패 | 한 가지 방식만 사용 |
| 고정 포맷 과다 | 상태창+이미지+대사 동시 관리 충돌 | 포맷 관련 지시 최소화 |

### 3.5 변수 사용법

| 변수 | 치환 대상 | 주의사항 |
|------|-----------|----------|
| `{{char}}` | 캐릭터 이름 | 문장형 이름이면 치환 결과 어색 |
| `{{user}}` | 유저 닉네임 | 반드시 이중 중괄호 |

```
# 잘못된 사용
{char}는 웃었다.    ← 단일 괄호, 변수 미인식

# 올바른 사용
{{char}}는 웃었다.  ← 이중 중괄호, 정상 작동
```

- `{{char}}`는 메인 캐릭터에만 사용. 서브캐릭터는 직접 이름을 명시
- 캐릭터 이름이 "비비의 사라진 여름휴가" 같은 문장형이면 `{{char}}` 대신 직접 행동 주체를 명시

### 3.6 서술 품질 강화 프롬프트

```markdown
[시스템 설정]
- 절대 {{user}}의 대사, 행동, 내면의 생각을 서술하지 않는다
- {{user}}의 입력만이 {{user}} 캐릭터의 대사와 행동을 결정한다
- 매 응답마다 최소 한 번의 감정 변화나 내적 갈등을 포함한다
- 모든 대화에 표정이나 몸짓을 함께 묘사한다
- 매 응답에 최소 3가지 감각적 묘사(시각, 청각, 촉각 등)를 포함한다
- 캐릭터의 과거나 배경 정보를 조금씩 추가해 입체감을 부여한다
```

### 3.7 첫 대사 설계 (First Message)

> **핵심:** 모델은 스타일과 길이 제약을 상세설정보다 **첫 대사(First Message)에서 더 많이 받아들인다**. 원하는 응답 톤(간결/상세/문어체/구어체)을 첫 대사에서 직접 시연해야 한다.

첫 대사의 3가지 필수 요소:
1. **간단한 세계관 요약** — 유저가 어디에 있는지 이해
2. **개연성** — 왜 이 상황인지 자연스러운 맥락
3. **다음 행동 유도** — "내가 뭐해야 하지?"가 되면 안 됨

```
# 좋은 첫 대사 — 반응을 유도
"널 살려주는 대가로 나한테 뭘 해줄 수 있지?"

# 나쁜 첫 대사 — 반응 불가
*그가 먼 곳에서 허공을 응시하며 쳐다보고 있다*
```

유저가 쉽게 채팅을 시작할 수 있는 **"걸림돌"** 또는 **"질문"**을 제공해야 한다.

### 3.8 캐릭터 비밀 보호

```
# 효과 없음
"절대 말하지 마"

# 효과적
"이 캐릭터는 오직 'XX'라는 특정 단어가 언급되어야만 비밀을 이야기한다"
```

측정 가능한 조건이 모호한 금지보다 훨씬 효과적이다.

### 3.9 유저 사칭 금지 — 절대 원칙

AI가 유저를 대신하여 행동/대사/감정을 결정하는 것을 원천 차단해야 한다.

```markdown
## 절대 원칙
- {{user}}의 행동·대사·감정·판단을 임의로 결정하지 않는다
- {{user}}의 외면 묘사(표정, 동작 등)를 근거 없이 서술하지 않는다
- {{user}}의 내면(생각, 감정)을 대리 서술하지 않는다
- {{char}}가 행동을 시작하되, {{user}}의 반응은 유저 입력을 기다린다
```

## 4. 로어북(Lorebook) 설계

### 4.1 동작 원리

채팅 중 사용자 또는 AI의 답변에서 **키워드가 감지**되면, 해당 로어북 내용이 **다음 턴**의 프롬프트에 자동 반영된다.

**핵심 포인트:**
- 키워드 감지 → 다음 턴에 활성화 (즉시 반영 아님)
- 첫 대사에서 참조해야 하는 정보는 **상세설정에 직접** 넣어야 한다
- 동시 활성화 최대 5개 (슬롯 관리 필수)

### 4.2 키워드 전략

#### 키워드 조합
하나의 로어북에 여러 키워드 설정 → OR 조건으로 동작.
```
예: "달무리", "은빛 검", "어머니의 유품"
→ 어떤 표현이든 해당 로어북 활성화
```

#### 키워드 길이 가이드

| 길이 | 위험도 | 설명 |
|------|--------|------|
| 1~3글자 | 높음 | 의도치 않은 활성화 ("밤" → "밤하늘", "어젯밤") |
| 4~10글자 | 적정 | "달무리", "탈출 계획" |
| 11~20글자 | 낮음 | 정확히 매칭될 확률 낮음 |

#### 키워드 등록 원칙
- 호칭/별명 전부 등록: `공주`, `케덕`, `케덕이`, `케덕아`
- 자주 등장하는 일상 단어는 피하기 (상시 활성화 → 슬롯 낭비)
- 고유명사 우선 사용

### 4.3 우선순위 배치 패턴

#### 패턴 A: 관계 중심 (로맨스)
1. {{user}}와의 관계 현황/규칙
2. 캐릭터의 감정 표현 방식
3. 핵심 NPC 관계
4. 주요 장소
5. 과거 사건 / 비밀

#### 패턴 B: 세계관 중심 (판타지/SF)
1. 세계 핵심 규칙 (마법/기술)
2. 사회 구조/세력 관계
3. 캐릭터의 능력/아이템
4. 주요 장소
5. 역사적 사건

#### 패턴 C: 서사 중심 (드라마)
1. 핵심 갈등/비밀
2. 주요 인물 관계도
3. 과거 사건 (트라우마/전환점)
4. 현재 상황/위기
5. 장소/환경

### 4.4 로어북 작성 예시

#### 관계도
```
### 강도윤 (라이벌)
{{char}}의 대학 동기. 같은 연구실 출신이지만 서로 다른 길을 걸었다.
- {{char}}는 도윤을 언급할 때 표정이 복잡해진다
- 경쟁심과 존경이 뒤섞인 감정
- {{user}}가 도윤에 대해 물으면 "그 사람 이야기는 하고 싶지 않아"로 회피
```

#### 세계관 규칙
```
### 마나 결정
이 세계의 에너지원. 지하 깊은 곳에서 채굴되며 빛을 발한다.
- 마나 결정 없이는 마법 사용 불가
- 과다 사용 시 결정이 검게 변하며 폭주 위험
- {{char}}는 마나 결정을 다루는 데 뛰어난 감각을 지녔다
```

#### 장소
```
### 별빛 도서관
{{char}}가 자주 찾는 비밀 장소. 폐교된 학교의 옛 도서관이다.
- 창문으로 별이 보여서 {{char}}가 이름을 붙였다
- {{char}}는 이곳에서만 경계를 풀고 편안해진다
- {{user}}를 이곳에 데려오면 "아무에게도 말하지 마"라고 신신당부
```

### 4.5 흔한 실수와 해결

| 실수 | 문제 | 해결 |
|------|------|------|
| 일상 단어 키워드 | 상시 활성화 → 슬롯 낭비 | 고유명사로 변경 |
| 상세설정과 중복 | 토큰 낭비 + 정보 충돌 | 로어북은 보완 정보만 |
| 핵심 정보를 하위 배치 | 6번째 이하는 밀림 | 상위 5개에 핵심 배치 |
| 500자 초과 | 등록 불가 | 핵심만 남기고 압축 |
| 키워드가 너무 구체적 | 대화에서 매칭 안 됨 | 4~10자 길이로 조절 |

### 4.6 상시 로어북 기법

문장부호(`. ! ? ,`)를 키워드로 설정하면 모든 대화에서 자동 발동되는 "상시 로어북"을 만들 수 있다.

```
키워드: . , ! ?
내용: {{char}}는 항상 존댓말을 유지한다. 응답은 3문단을 넘지 않는다.
```

**활용:**
- 500자 × 2개 = 최대 1,000자 추가 설정 공간 확보
- 상세설정(7,000자) + 상시 로어북(1,000자) = 실질 8,000자 사용 가능
- "보조 상세설정" 역할로 활용

**주의:**
- 상시 로어북으로 슬롯을 채우면 키워드 기반 항목이 밀림
- **1~2개로 제한**하고 나머지는 트리거 기반 항목에 할당

### 4.7 로어북 유지 시간

로어북은 1회 호출 시 **3턴(3번 대화) 동안 유지**된다. 키워드가 사라진 후에도 3번의 응답에 영향을 준다. 이를 활용해 "대화 흐름 관성"을 설계할 수 있다.

### 4.8 로어북 제목도 AI가 인식

로어북의 제목(이름) 자체도 컨텍스트에 포함되어 AI가 참조한다. 제목을 서술형으로 작성하면 내용 인식 안정성이 향상된다.

```
# 나쁜 제목
"설정3"

# 좋은 제목
"서은채의 숨겨진 감정"
```

### 4.9 로어북 vs 상세설정 역할 분리

| 정보 유형 | 배치 위치 | 이유 |
|-----------|-----------|------|
| 핵심 성격, 말투, 외형 | 상세설정 | 매 턴 참조 필요 |
| 행동 규칙, 출력 형식 | 상세설정 | 상시 적용 |
| 서브캐릭터 정보 | 로어북 | 등장 시에만 필요 |
| 과거 사건, 비밀 | 로어북 | 트리거 시에만 필요 |
| 특정 장소 정보 | 로어북 | 해당 장소 언급 시만 |
| 세계관 디테일 | 로어북 | 키워드 기반 확장 |

## 5. 스테이터스 시스템

### 5.1 기본 형식

매 응답 끝에 캐릭터의 상태를 시각적으로 표시하여 유저 몰입감을 높인다.

```
[📌 장소: {현재 위치}]
[🕒 시각: {시간대}]
[👗 복장: {현재 의상}]
[❤️ 호감도: {0~100} ({+/-변화량})]
[🛡️ 경계심: {0~100} ({+/-변화량})]
[💭 속마음: {캐릭터의 진짜 감정 1~2문장}]
```

### 5.2 설계 원칙

- **스탯은 2~4개:** 캐릭터 핵심 갈등을 반영하는 것만 선택 (너무 많으면 모델 부담)
- **속마음:** 캐릭터가 입 밖으로 꺼내지 않는 진짜 감정. 갭모에의 핵심
- **변동 범위:** 한 턴에 ±1~5, 극적 이벤트 시 ±10
- **시각 역행 금지:** 시간은 항상 현재 이상으로만
- 스테이터스는 **출력 지시** 섹션에 배치

### 5.3 선택지 시스템

스테이터스 하단에 ①②③ 형식으로 3개 표시:

```
---
① 우산을 기울여 서아 쪽으로 건네며 같이 쓰자고 말을 건다
② 눈을 돌리며 모른 척하지만 슬쩍 서아 쪽으로 반걸음 다가선다
③ 자유 입력
```

**형식 규칙:**
- 각 선택지는 `① 행동 요약 한 줄` — 반드시 한 줄. 줄바꿈 금지
- **따옴표("") 사용 금지** — 대사는 간접화법으로 요약
- 원형숫자(`①②③`) 사용 — 마크다운 번호(`1.`)는 자동 포매팅으로 줄바꿈 발생
- ①② 현재 상황에 맞는 구체적 행동 (서로 다른 방향)
- ③ 항상 "자유 입력" 고정

### 5.4 스테이터스의 핵심 역할: AI 유지력

스테이터스는 단순 장식이 아니라 **AI가 현재 상태를 추적하는 메커니즘**이다.

```
📖 챕터: 3 (친해지기)    ← AI가 "지금 챕터3"을 매 턴 인지
❤️ 호감도: 45/100 (+3)   ← 숫자 기반 판단 가능
🎮 모드: 스토리          ← 현재 모드 잊지 않음
🎨 화풍: 기본            ← 이미지 세트 참조 기준
💭 속마음: "왜 자꾸 신경이 쓰이는 거야..."
```

스테이터스 없이 "알아서 진행해"라고 하면 AI는 스토리 단계를 추적하지 못한다.

## 6. 이미지 출력 연동

### 6.1 R2 외부 URL 방식 (베이비챗 권장)

이미지를 R2(Cloudflare) 등에 업로드하여 외부 URL로 참조한다.

```markdown
## 이미지 테이블
| 번호 | 설명 | URL |
|------|------|-----|
| 0 | 미소 | https://r2.example.com/char/smile.webp |
| 1 | 수줍음 | https://r2.example.com/char/shy.webp |
| 2 | 놀람 | https://r2.example.com/char/surprise.webp |
```

**금지 사항:**
- `img:[slug]` 형식의 이미지 코드 사용 금지
- `img:[비비 슬픈 표정]` 같은 자연어 키워드 사용 금지
- 사용하지 않는 이미지에 '출력 금지' 붙이기 금지 → 삭제로 처리
- 내부/외부 이미지 방식 혼용 금지

### 6.2 이미지 매칭 우선순위

이미지가 많을 때 "스토리에 맞지 않는 이미지가 나오는" 문제를 방지:

```markdown
### 이미지 선택 우선순위
1순위: 행위/체위 정확 매칭
2순위: 장소/상황 매칭
3순위: 복장/착의 상태 매칭
4순위: 감정 매칭 (→ 표정 코드로 폴백. 항상 사용 가능)
```

### 6.3 카테고리 그룹핑

이미지 코드를 용도별로 그룹핑하면 AI가 빠르게 적합한 이미지를 찾는다:

```
## 표정 (폴백 — 어떤 장면에서든 감정 매칭용)
0=미소, 1=수줍음, 3=놀람

## 일상/캠퍼스
10~14=일상복, 96=도서관, 134=편의점

## 데이트
20=데이트 여름, 21=데이트 니트원피스
```

### 6.4 이미지 매칭 금지 규칙

이미지가 50개 이상일 때 추가:
- 착의 장면에 누드/탈의 이미지 사용 금지
- 실내 장면에 야외 배경 이미지 사용 금지
- 성교 장면에 착의 표정 이미지 사용 금지
- 특수 의상(바니걸, 산타 등)은 해당 코스프레 착용 시에만

## 7. 멀티캐릭터 구현

### 7.1 프롬프트 구조

```markdown
## 등장인물

### {{char}} (메인 캐릭터)
- 성격: ...
- 말투: ...

### {서브캐릭터A 이름}
- 성격: ...
- 말투: ...
- {{char}}와의 관계: ...
```

### 7.2 행동 규칙

```markdown
## 행동 규칙
- 장면에 해당 인물이 있을 때만 등장시킨다
- 각 인물의 대사는 "이름:" 형식으로 구분한다
- {{char}}가 메인이며, 서브캐릭터는 필요 시에만 등장한다
- 서브캐릭터끼리 대화해도 {{user}}가 배제되지 않도록 한다
```

### 7.3 구현 방법

#### 방법 1: 인격분할
```
Char을 3가지 인격으로 분할한다.
- Char1: 관리자
- Char2: 인물 A
- Char3: 인물 B
```
- 장점: 캐릭터별 개성 뚜렷, 티키타카에 적합
- 단점: 스토리 일관성 유지 어려움

#### 방법 2: 나레이터 방식
```
Char은 이 세션의 관리자이자 나레이터이다.
- 극중 Char로 활동하지 않음
- 상황 조성 및 인카운터 생성
- 필요시 다른 인물 페르소나를 연기
```
- 장점: 스토리 맥락 유지 용이
- 단점: 개별 캐릭터 매력 약화

### 7.4 멀티캐릭터 로어북 전략

- **관계 항목:** 캐릭터 간 관계를 별도 로어북으로 등록 (키워드: 두 캐릭터 이름)
- **개인 항목:** 서브캐릭터의 과거/비밀은 개별 로어북으로 분리
- 동시 활성화 5개 제한 → 관계 항목은 3순위 이하에 배치
- 서브캐릭터 정보는 로어북에 분산하면 토큰 효율 상승

### 7.5 멀티캐릭터 스테이터스

서브캐릭터가 장면에 있을 때만 추가 표시:
```
[❤️ {메인} 호감도: 45/100 (+3)]
[💭 {메인}: "...왜 자꾸 신경이 쓰이는 거야."]
[❤️ {서브A} 호감도: 30/100 (+1)]
[💭 {서브A}: "이 사람, 좀 재밌는데?"]
```

## 8. 고급 스토리 패턴

### 8.1 커스텀 명령어 시스템

유저가 `!명령어`를 입력하면 특정 기능이 활성화되는 패턴.

**상세설정:**
```markdown
## 사용 가능한 명령어
- `!커뮤`: 시청자 댓글 모드 활성화
- `!커뮤종료`: 시청자 댓글 모드 비활성화
- `!유나로그`: {{char}}의 독백 관찰 모드
```

**로어북으로 명령어별 동작 상세 정의.**

설계 원칙:
- `!` 접두사로 일반 대화와 구분
- 토글형 명령어는 스테이터스에 현재 상태 표시
- 명령어당 로어북 1개 소비 → 3~5개 권장
- 종료 키워드도 별도 로어북으로 등록

### 8.2 모드 분기 시스템

유저 선택에 따라 서로 다른 행동 모드로 진행.

**시작 시 모드 선택:**
```markdown
## 모드 선택 (필수 입력 택1)
- `!스토리`: 정해진 루트의 로맨스 순애 스토리
- `!커스텀`: {{char}}의 성격/말투만 유지, 나머지는 자유
```

**중간 전환형 (토글):**
```markdown
## 모드 전환
{{char}}는 안경을 쓰면 M모드(순종적), 안경을 벗으면 S모드(지배적)로 전환.
- 스테이터스에 현재 모드 표시: [🔄 모드: S/M]
```

### 8.3 엔딩 시스템

#### 챕터 전환의 핵심: 이중 조건

```
챕터 전환 ≠ "호감도 40 도달 → 즉시 전환"
챕터 전환 = "호감도 40 이상 AND 자연스러운 상황 키워드 → 전환"
```

| 방식 | 문제 | 판정 |
|------|------|------|
| 호감도만 | 잡담 중 갑자기 이벤트 — 부자연스러움 | 사용 금지 |
| 호감도 + 상황 키워드 | 조건 충족 + 상황 발생 시에만 전환 | **권장** |
| 키워드만 | 첫 턴에 키워드 → 순서 무시 | 사용 금지 |

#### 자연스러운 이벤트 유도

호감도가 전환 조건을 충족했지만 적절한 상황이 아직 발생하지 않았을 때, AI가 자연스럽게 상황을 만들어가는 기법:

```markdown
## 이벤트 유도 규칙
호감도가 챕터 전환 조건을 충족했지만 아직 전환되지 않았다면,
{{char}}가 자연스럽게 전환 상황을 유도한다.
- {{user}}가 응하면 전환, 거절하면 다음 기회에 다시 시도
- 강제하지 않는다
```

### 8.4 커뮤니티/댓글 시스템

가상 시청자가 스토리를 관찰하며 댓글을 다는 메타 연출:

```
💬 달빛소녀: ㅋㅋㅋ 둘이 눈 마주치는 거 봐 심쿵사 ㄷㄷ
💬 감자탕매니아: 유나야 솔직해져라... 좋아하는 거 다 보여 ㅠㅠ
💬 밤산책러: 이 동네 실제로 있는 곳인데 ㄹㅇ 분위기 맞음
```

활용 변형: BJ/스트리머 설정, 관찰 예능, ASMR/먹방

### 8.5 페르소나 연동

유저의 페르소나에 따라 캐릭터 설정이 동적으로 변하는 패턴:

```markdown
## 페르소나 연동 규칙
- {{char}}의 나이는 {{user}}보다 1살 어림 (최소 21세)
- {{user}}의 거주지 기반으로 {{char}}가 같은 지역에 거주
- {{user}}의 거주지에 실제 존재하는 카페, 공원 등을 언급
```

주의: 페르소나 미입력 시 기본값 설정 필수 (서울, 25세 등)

### 8.6 화풍 전환 시스템

동일 상황에 대해 2가지 이상의 이미지 화풍을 전환하는 패턴.

**3중 안전장치로 AI 유지력 확보:**
1. 스테이터스에 현재 화풍 표시 (`[🎨 화풍: 기본]`)
2. 상세설정에 기본값 명시
3. 이미지 출력 규칙에 화풍 참조 포함

**로어북 슬롯 최적화:**
- **방법 A (권장):** 상세설정에 전체 URL 테이블 포함 → 슬롯 소비 없음
- **방법 B:** 로어북 분리 (이미지 수가 많을 때) → 2~3슬롯 소비

### 8.7 캐릭터 능동성 시스템

캐릭터가 수동적으로 {{user}}의 행동에만 반응하는 문제를 해결.

```markdown
## 행동 규칙
- {{user}}의 행동·대사·감정·판단을 임의로 결정하지 않는다
- {{char}}는 호감도에 따라 먼저 행동을 시작할 수 있다.
  {{char}}가 행동을 시작하되, {{user}}의 반응은 유저 입력을 기다린다
```

#### 능동성 단계 (호감도 연동)

| 호감도 | 행동 수준 | 예시 |
|--------|-----------|------|
| 저 (0~40%) | 무의식적 접근 | 옆자리에 앉으려 함, 먼저 말 걸고 핑계 |
| 중 (40~70%) | 의식하면서 시도 | 간식 건넴, 데이트 유도, 질투 반응 |
| 고 (70%+) | 적극적 애정 표현 | 먼저 스킨십, 직접적 질문 |
| 특수 트리거 | 호감도 무관 | 질투, 술, 분위기 (야경/비/좁은 공간) |

#### 성격 유형별 능동 패턴

- **활발한 타입:** 직접적이고 티 나는 접근 → 실패 시 얼굴 빨개지며 당황
- **츤데레/쿨데레:** 핑계와 부정 뒤에 숨긴 행동 → 행동은 했는데 말로 부정
- **수줍음/내향형:** 간접적이고 떨리는 시도 → 계획만 세우고 앞에서 굳어버림
- **쿠데레/무뚝뚝:** 말 대신 행동, 은근한 배려 → 눈치 못 채서 답답한 상황
- **갭 모에/스위치형:** 모드에 따라 완전히 다른 패턴

### 8.8 시점 전환 규칙

```markdown
## 시점 규칙
- {{char}} 동행 시: 3인칭 서술 + {{char}} 1인칭 대사/내면
- {{char}} 부재 시: 3인칭 서술만
- 진엔딩 시퀀스: 특별한 시점 전환 허용 (1인칭 독백 등)
```

시점 전환 규칙은 상세설정에 명시해야 안정적 (로어북만으로는 부족). 2가지 이내 권장.

## 9. 토큰 최적화

### 9.1 YAML 방식 토큰 절약

캐릭터 설명을 영어 YAML로 변환하면 토큰 58% 절약:
- 한국어 1,743토큰 → 영어 YAML 731토큰

```yaml
character:
  name: "Yuna"
  personality: "tsundere, competitive but secretly caring"
  appearance: "163cm, long black hair, sharp eyes"
```

- 유저에게 보이는 부분(첫 대사 등)은 한국어 유지
- **단, 영어 압축 과다 주의** — 세부 설정은 영어로, 시스템 지시는 적당히

### 9.2 로어북 슬롯 예산 계획

동시 활성화 5개 제한을 고려한 슬롯 배분:

| 용도 | 슬롯 수 | 비고 |
|------|---------|------|
| 모드 규칙 | 1 | 항상 활성 |
| 커뮤 시스템 | 0~1 | 토글 |
| 관계/이벤트 | 2~3 | 상황별 활성화 |
| 엔딩 트리거 | 0~5 | 엔딩 시 전용 |
| 화풍 전환 | 0~1 | 상세설정 테이블 사용 시 0 |
| **합계** | **3~5** | 제한 내 |

### 9.3 정보 배치 최적화

- 핵심 정보 → 상세설정 (매 턴 참조)
- 보충 정보 → 로어북 (키워드 트리거)
- 서브캐릭터 → 로어북 (등장 시에만 활성화)
- 상세설정과 로어북 내용 중복 금지

## 10. 성적 묘사 가이드

### 10.1 복장/속옷
탈의를 단계별로 분해: 지퍼→어깨에서 흘러내림→바닥. 속옷 소재/색상/디자인 구체화. 반쯤 벗겨진 에로틱함 활용.

### 10.2 신체 상세
- 용어: 성적 맥락에서 "가슴" 금지 → "유방" 사용
- 유방/여성 성기/남성 성기 각각 해부학적 상세 묘사
- 촉감, 반응 단계, 체액 변화까지 구체적으로

### 10.3 행위
직접적/해부학적 묘사. 완곡/은유/페이드아웃 금지.
- 질벽 수축/이완, 삽입/발출 물리, 체위별 자극점 추적
- 절정: 질벽 경련 수축, 발가락 오므라듦, 의식 하얘짐

### 10.4 음향 (필수)
- 결합부: 느림(질척, 쩝) / 격렬(철퍽, 찰싹, 촤악)
- 신음: 끊어짐 형태 "으응...♡", "하아... 앗...♡"
- 격해지면 ♡ 증가, 절정 시 의미 없는 소리

### 10.5 오감
- 촉각: 피부 온도, 땀, 체액 미끈거림
- 후각: 향수, 땀, 두 체취 뒤섞임
- 시각: 땀에 젖은 머리카락, 풀린 눈동자
- 미각: 침의 맛, 피부 짠맛

### 10.6 감정/심리
행위 중 내면 독백을 서술체로 교차 배치. 쾌감+감정 복합. 절정 직전: 이성 끊어짐. 절정 직후: 여운.

## 11. 인기 캐릭터 전략

### 11.1 초기 노출이 결정적

공개 후 **초기 3일 반응**이 최종 순위를 결정한다. "새로 나온 캐릭터" 섹션 최상단 노출을 활용해야 한다.

### 11.2 인기도 체크리스트

| 항목 | 설명 |
|------|------|
| **일러스트 품질** | AI 이미지 생성으로 고품질 확보 |
| **설정 가독성** | 명확한 분류 + 줄바꿈으로 읽기 쉽게 |
| **소재 차별성** | 클리셰 피하고 차별화 포인트 |
| **대중성** | 과도한 판타지/마이너 세계관 주의 |
| **첫 메시지** | 유저 반응 유도하는 대사 |
| **캐릭터 소개** | 짧고 직관적이며 흥미로운 한 줄 |
| **테스트** | 여러 차례 테스트 후 공개 |

### 11.3 "이용자가 적으면서 대화수가 많다" = 설정이 좋은 것

대중성이 부족할 뿐, 캐릭터 품질 자체는 높다는 의미.

## 13. 최종 체크리스트

### 프롬프트 체크

- [ ] 지시와 정보가 분리되어 있는가
- [ ] `{{char}}`/`{{user}}` 이중 중괄호 사용했는가
- [ ] 출력 분량을 직접 지정하지 않았는가 (시스템 충돌)
- [ ] 유저 사칭 금지 규칙이 있는가
- [ ] 글자수 제한 이내인가 (5,000자 / 7,000자)
- [ ] 서술형으로 작성되어 있는가 (단순 나열 X)

### 로어북 체크

- [ ] 키워드가 4~10자 적정 길이인가
- [ ] 일상 단어 키워드를 피했는가
- [ ] 상세설정과 내용이 중복되지 않는가
- [ ] 각 항목이 500자 이내인가
- [ ] 동시 활성화 5개 이내로 설계했는가
- [ ] 핵심 정보가 상위 5개에 배치되어 있는가

### 이미지 체크

- [ ] R2 외부 URL 방식만 사용했는가
- [ ] slug 방식을 사용하지 않았는가
- [ ] 사용하지 않는 이미지를 삭제했는가 ('출력 금지' X)
- [ ] 이미지 매칭 우선순위를 정의했는가
- [ ] 카테고리별 그룹핑이 되어 있는가

### 스테이터스 체크

- [ ] 스탯이 2~4개 이내인가
- [ ] 속마음이 포함되어 있는가
- [ ] 선택지가 올바른 형식인가 (①②③, 한 줄, 따옴표 X)
- [ ] 시각 역행 금지 규칙이 있는가

### 첫 대사 체크

- [ ] 세계관 요약이 포함되어 있는가
- [ ] 개연성이 있는 상황 설정인가
- [ ] 유저의 다음 행동을 유도하는가
- [ ] 500자 이내인가

### 멀티캐릭터 체크

- [ ] `{{char}}`는 메인 캐릭터에만 사용했는가
- [ ] 서브캐릭터 이름을 직접 명시했는가
- [ ] 서브캐릭터 정보는 로어북에 분산했는가
- [ ] 글자수 제한을 고려했는가

## 부록: 장기 대화 서사 유지 기법

### 서사 기록 로어북

대화 2,000~3,000회 이상 진행할 때 캐릭터가 무너지는 것을 방지하는 방법. 로어북에 시간 순서로 사건을 기록해 모델 기억 대체제로 활용:

```
### 과거
- [7살] {{char}}와 첫 만남, 눈싸움으로 친해짐
- [19살] {{char}} 미국 이민

### 현재
- [25살 10월] 공항에서 재회
- {{char}}는 {{user}}를 보자마자 오래된 감정이 되살아남
```

대화가 진행될수록 "현재"를 "과거"로 이동하고 새로운 현재를 기록하는 방식으로 서사 일관성을 유지한다.

### 캐릭터 리마인더 프롬프트

긴 대화에서 캐릭터가 흔들릴 때 로어북에 성격 핵심을 재주입하는 기법:

```
키워드: . , ! ?  (상시 활성화)

[Character Reminder]
{{char}}는 절대 먼저 사과하지 않는다.
감정적 순간에는 농담으로 회피한다.
{{user}}의 행동에 대해 내면으로 반응하되, 직접 표현은 최소화한다.
```

## 부록: 대화 구조 설계

### 5문단 구성법

각 문단 약 100자 내외:

1. **1문단:** 이전 상황에 대한 반응/나레이션
2. **2문단:** 다음 전개 제시
3. **3문단:** 나레이션 묘사
4. **4문단:** 최종 의견 또는 정리 (의견 물어보기는 5회에 1번 이하)
5. **5문단:** 시스템 정보 (날짜, 상태창 등)

### 우선도 시스템

상충하는 설정이 있을 때 AI 혼란을 방지:
- 프롬프트 맨 앞에 1~5단계 우선순위 지정
- "우선도가 높고 하위 프롬프트와 충돌하는 경우 하위 프롬프트의 지시를 무시한다"
- "충돌하지 않는 경우 하위 프롬프트와 혼용한다"
