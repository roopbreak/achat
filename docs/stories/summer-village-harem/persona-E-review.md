## FAIL (수정 필수)

### F-1. [D/C] `!command` 트리거 로어 scan_depth=4 — 모드 오염 4턴 잔존

**현재**: 모드 트리거 로어 [1449]~[1454] 6개 전부 본문 텍스트에는 `**!고향모드 진입 — scan_depth: 1**`이라 적혀 있으나, **DB 실제 `scan_depth` 컬럼 값은 4**(다이제스트 헤더 `scan_depth 4` 및 `lib/db.mjs:117` `ALTER TABLE lore_entries ADD COLUMN scan_depth INTEGER DEFAULT 4` — 명시 안 하면 4로 들어감).

**코드 확증** — `lib/context-builder.mjs:257-267`:
```js
const depth = e.scan_depth ?? 4;
const scanTexts = recentMsgs.slice(-(depth * 2)).map(m => m.content);
const combined = [...scanTexts, userInput].join(' ');
return keywordMatch(combined, e.keys);
```
`depth=4` → `slice(-8)` → 최근 8개 메시지(= 4턴) + userInput을 스캔. 유저가 `!고향모드`를 입력한 메시지는 그 후 **4턴 동안** 스캔 윈도우에 잔존 → [1449] 로어가 4턴 내내 재주입된다. [1449] 본문에는 `첫 마중 권장: 최인혜(이장)가 마중 + "어머 너 ○○이 맞지?"` 같은 **일회성 진입 지시**가 들어 있어, 4턴간 재주입되면 AI가 "첫 마중"을 반복하거나 진입 톤을 계속 재적용한다. 모드의 지속 규칙(호감도 시작값·톤)은 이미 상시 로어 [1433] 모드 시스템에 있으므로 [1449]~[1454]의 역할은 *진입 1회 트리거*뿐 — 4턴 잔존은 순수 오염.

`qa-checklist.md` D: `명령어 트리거 | !command 인데 scan_depth≠1` → **FAIL**.

**수정안**: [1449]~[1454] 6개 엔트리의 DB `scan_depth`를 **1**로 변경. (참고: 코드 구조상 `slice(-(depth*2))`이라 scan_depth=1도 `slice(-2)`로 직전 1턴 bleed가 있으나, 4 대비 잔존이 4턴→1턴으로 줄어듦.) `updateLoreEntry(id, { scan_depth: 1 })` 또는 재임포트 시 엔트리에 `scan_depth: 1` 명시.

### F-2. [H/D] 모드 트리거 키워드 substring 충돌 — `!음란고향모드` 입력 시 `!고향모드` 로어 동시 발화

**현재**:
- [1449] `!고향모드` keys: `["!고향모드","고향모드"]`
- [1453] `!음란고향모드` keys: `["!음란고향모드","음란고향모드"]`

**코드 확증** — `lib/context-builder.mjs:230-252` `keywordMatch`는 순수 substring 매칭(`lower.includes(trimmed.toLowerCase())`). 유저가 `!음란고향모드`를 입력하면:
- `"!음란고향모드".includes("고향모드")` → **true** ([1449]가 매칭됨)
- `"!음란고향모드".includes("음란고향모드")` → true ([1453]도 매칭됨)

→ `!음란고향모드` 한 번 입력에 **[1453] + [1449] 두 진입 로어가 동시 주입**된다. AI는 "고향모드"와 "음란고향모드" 진입 지시를 동시에 받아 톤/호칭/마중 인물이 충돌한다.

`qa-checklist.md` D: `키워드 형태 — 일상어·범용어` 이상의 구조적 결함 + H `로어북이 적절한 타이밍에 트리거` 위반 → **FAIL**.

**수정안**: 모드 키워드를 substring 충돌이 없도록 재설계.
1. 권장: 키를 `!` 포함 명령어 1개로만 좁히되 충돌 없는 형태로 — [1449] keys `["!고향모드"]`, [1453] keys `["!음란고향모드"]` (접미사 "모드" 제거형 `"고향모드"` 같은 부분 키 삭제).
2. `!음란고향모드`가 `!고향모드`를 포함하는 구조 자체가 위험 → 모드 명령어를 비포함 관계로 리네이밍 검토(`!고향`, `!음란고향`, `!귀농` … 이어도 `!음란고향`⊃`고향` 미발생 확인). 단 이건 first_mes·post_history·[1433] 안내문까지 일괄 수정 필요.
3. 최소 수정안만 적용한다면 (1) — 부분 키 `"고향모드"`/`"음란고향모드"` 등 접미어 키를 전부 제거하고 `!` 풀네임 키만 남긴다.

### F-3. [C] 내부/외부 이미지 URL 혼용 + 두 포맷 충돌

**현재** — 같은 스토리 안에서 이미지 URL 포맷이 **두 개** 공존:

2. `lib/context-builder.mjs:362` → `imageSection = hasImageMapping ? '' : buildImageSection(...)` — 즉 **엔진의 정식 내부 이미지 카탈로그(buildImageSection) 주입이 차단된다.** AI는 `story_images`에 실제로 어떤 scene_key가 있는지 못 보고, [1430] 상황 코드 매핑(01~75)을 보고 추측해서 코드를 찍는다.
3. `routes/images.mjs` + `index.mjs:50` `app.use('/images', imagesRouter)` — `risu.ddsmdy.com/images/{story}/{charDir}/{sceneKey}`는 **외부 URL이 아니라 이 서버 자신의 내부 라우트**다. `serveImage()` → `getRandomImage(storyName, charDir, sceneKey)`로 `story_images` 테이블에서 파일을 찾아 디스크에서 서빙한다. 즉 description은 **내부 경로**, [1429] 로어는 **외부 R2 버킷** — 완전히 다른 스토리지.

`achat-engine.md` §11: `❌ 내부/외부 이미지 혼용` — 명시적 위반. → **FAIL**.

추가 위험: description은 risu.ddsmdy.com 경로를 "외부 URL"이라 표기("## 이미지 시스템 (외부 URL — 자체 이미지 시스템 미사용)")하지만 실제로는 자체 서버 내부 라우트 → 작성자가 시스템 구조를 오인한 상태. 어느 쪽이 실제로 200을 주는지(`story_images` 테이블에 행이 있는지 / R2 버킷에 객체가 있는지)는 **서버 확인 필요** — 그러나 두 포맷이 system prompt에 동시에 들어가는 것 자체가 결함(엔진은 둘 다 검증 없이 텍스트로 통과시킬 뿐).

**수정안**: 이미지 소스를 **하나로 단일화**.
- (A) 내부 라우트 유지: [1429] 로어와 description 본문을 모두 `![](https://risu.ddsmdy.com/images/{스토리}/{캐릭터}/{코드})` 포맷으로 통일하고, `story_images` 테이블에 캐릭터별 01~75 행이 실제 있는지 검증. 단 이 경우 `hasImageMapping=true`로 정식 카탈로그가 막히므로 차라리 description에서 `![](http...)` 리터럴 예시를 제거해 `hasImageMapping=false`로 만들고 엔진 `buildImageSection` 자동 카탈로그를 쓰는 게 정석.

- 어느 쪽이든 **두 포맷 공존을 제거**하는 것이 핵심.

## WARN (개선 권고)

### W-1. [D] 상시 로어 `insertion_order`가 9개 전부 100 — 우선순위 의도 무력화

**코드 확증** — `lib/db.mjs:365` `getConstantLore` → `ORDER BY insertion_order` (priority 미사용, 2차 정렬 키 없음). 다이제스트상 9개 상시 로어 모두 `io 100`. SQLite는 동순위 tie를 idx_lore_story 스캔 순(≈rowid/id 순)으로 반환 → 실제 주입 순서는 id순(1429→1437)이고, 다이제스트가 보여주는 priority 정렬(100→95→90)은 **엔진에 반영되지 않는다**. 즉 상시 로어의 `priority` 필드는 사실상 dead. 한 세션 내에선 순서가 안정적이라 캐시는 깨지지 않으나, 의도한 배치(안전장치·INFO를 앞에)가 보장되지 않음.

**수정안**: 상시 로어 압축(F-4) 후 남는 항목에 distinct `insertion_order` 부여(예: 10/20/30…), 안전장치·INFO 형식을 앞 순번으로.

### W-2. [C] "며칠 안 보임"(민지유 [1445]) — 시간(일수) 카운팅 의존

[1445] 본문: `위협 신호: 너무 들이대다 거절당하면 자존심 상처 → 며칠 안 보임`. AChat은 턴/일수 카운터가 없고(`achat-engine.md` §8 "턴 수 기반 조건 절대 금지 — AI는 턴을 세지 못한다"), INFO 박스의 날짜는 AI가 수동 진행. "며칠"이라는 정량 간격은 AI가 신뢰성 있게 유지 못 한다. 구조적 페이즈 게이트가 아니라 캐릭터 플레이버 텍스트라 FAIL은 아니나 동작 보장 안 됨.

**수정안**: 이벤트 기반 서술로 치환 — "다음 만남에서 데면데면하게 굴고 먼저 말 걸지 않음" 처럼 *간격*이 아니라 *상태/행동*으로 표현.

### W-3. [H] INFO 박스 관계 횟수(💦) 누적 카운터 — 유지 불가

[1431] `💦{관계 횟수}` "캐릭터별 누적 성관계 횟수, 0부터 시작". AI는 단조 증가 카운터 유지에 약하고, `lib/db.mjs:590` `getActiveMessages`는 `summarized=0` 메시지만, `lib/context-builder.mjs:489` `recent`는 그중 최근 `RECENT_TURNS(8)*2`개만 본문에 넣는다. 카운터 값이 8턴 윈도우 밖으로 밀려나고 세션 요약에 안 잡히면 AI는 직전 INFO 박스를 못 봐 값을 잃거나 리셋한다. 11명 × 카운터면 더 취약. 호감도(❤️)는 0~200/200~500… 밴드 설명이 앵커라도 되지만 💦는 앵커가 없는 순수 기억 의존.

**수정안**: 💦 누적 카운터 제거(F-6과 함께). 굳이 둔다면 "관계 횟수" 대신 관계 *단계*("첫 관계 전 / 관계 있음 / 깊은 관계")처럼 밴드화 — 드리프트가 치명적이지 않게.

### W-4. [B] post_history_instructions 과다 길이

현재 post_history는 "[응답 작성 순서]" 7항목 + "[모드별 톤 요약]" 6항목 + "[합의 안전장치]" 4항목 + "[참조]" 6항목 ≈ 25줄+ / ~1,250자. `achat-engine.md` §3 권장 "3~5줄". 매 턴 마지막 재주입 블록이 길면 핵심 규칙이 흐려진다. F-6 수정안에 포함.

### W-5. [D] 로어 항목 수 33개

### W-6. [E/D] 장소·사물 로어 ↔ 바베군 세계관 상시 로어 중복

[1455] 수박·[1456] 물놀이·[1459] 회관·[1460] 비닐하우스·[1461] 우물의 감각 디테일은 상시 로어 [1434] 바베군 세계관 "감각 디테일" 항목과 실질 중복. F-5의 오매칭으로 이 로어들이 거의 매 턴 동적 블록(비캐시)에 올라오는데, 이미 캐시된 [1434]가 같은 내용을 담고 있어 **비캐시 토큰 예산을 중복 정보로 소모**. `qa-checklist.md` D `중복 — 부분 중복` WARN.

### W-7. [C] 한글 이미지 슬러그 — URL 동작 서버 확인 필요

이미지 URL 경로가 한글 캐릭터명(`최인혜`,`박경아`…)을 그대로 사용. 내부 라우트(`routes/images.mjs:27-34`)는 `decodeURIComponent(req.params.charDir)`로 디코드 후 `getRandomImage`로 `story_images.char_dir`와 매칭 — 브라우저가 와이어에서 percent-encoding하면 동작하나, **`story_images.char_dir`의 유니코드 정규화(NFC/NFD)가 요청과 일치해야 함**. `lib/db.mjs:152-194`의 NFD→NFC 마이그레이션은 `stories.name`·FK만 정규화하고 `story_images.char_dir`은 대상 아님 → char_dir이 NFD로 저장돼 있으면 NFC 요청과 불일치 404 위험. 외부 R2(`pub-xxx.r2.dev/최인혜/…webp`)는 업로드 시 객체 키 인코딩에 의존. 엔진(context-builder)은 이 URL을 검증 없이 텍스트 통과만 하므로 엔진측 FAIL은 아니나, **실제 fetch 200 여부는 배포 서버에서 확인 필요**. F-3 단일화 시 함께 검증.
