## 종합 판정: **FAIL**

FAIL 6건 (E 3 · A 2 · K 1) + Codex 신규 FAIL/격상 3건 + DB 실측 신규 1건 = **FAIL 10건**, WARN 다수.

핵심 원인은 **9개 모드-트리거 로어가 전부 "죽은 로어"**라는 것 — `!기본모드` 같은 키워드는 유저가 입력한 직후 ~4턴(scan window)만 트리거되고, 이후 상태창에는 `🐇` 이모지만 남아 트리거 문자열이 재등장하지 않으므로 **모드 정의·모드 효과가 컨텍스트에서 증발**한다. 이 스토리의 정체성(9개 모드로 분기되는 무인도 하렘)이 엔진 레벨에서 4턴 만에 무너진다. bangkok-poolvilla의 "죽은 Day 로어"와 정확히 같은 계열의 결함이다.

> **하네스 검증 메모**: 작성 시점 단일 QA(이전 03_qa_report.md)는 모드 시스템을 **PASS**로 판정하고 "9 모드 설계가 AChat 엔진 특성에 맞춰 잘 되어 있다"고 평가했다. 새 6인 하네스의 적대적 E 페르소나는 `matchLoreByKeyword`의 `scan_depth * 2` 윈도우 + `keywordMatch` substring 매칭을 코드로 확인해, 모드 효과가 세션 전체에 필요한데 모드-트리거 로어가 일회성 키워드에 묶여 죽는다는 것을 확증했다. 또한 단일 QA가 FAIL로 올린 "코드 미할당 캐릭터 404"는 **DB 실측(`GET .../images` 2,701장) 결과 f27~f30이 각 73장 풀세트로 존재**하여 사실관계가 틀렸음 — 「최서진 이루아」「아야카」가 부당하게 이미지 차단당하고 있었다. 반대로 새 하네스는 단일 QA·페르소나가 모두 놓친 **f6(윤정아) 이미지 실제 부재**(charDir에 f6만 없음 → 윤정아 이미지 삽입 시 진짜 404)를 DB 실측으로 발견했다. 다관점 + 적대적 E + 코드/DB 확증이 단일 시각의 사각지대를 양방향으로 메운 사례.

## FAIL (수정 필수)

### F1. [C·D·H / E·A] 9개 모드-트리거 로어가 전부 죽은 로어 🔴 최우선
`!기본모드`·`!하드모드`·`!음란모드`·`!TS모드`·`!NTR모드`·`!생존모드`·`!속마음모드`·`!연애서바이벌모드`·`!커스텀모드` 9개 키워드 로어는 모드 정의와 효과를 담고 있다. `matchLoreByKeyword`는 `recentMsgs.slice(-(scan_depth*2))` = 최근 8개 메시지 + 현재 입력만 substring 스캔한다. 유저가 1턴에 `!기본모드+!하드모드`를 입력하면 그 입력이 8메시지 윈도우에 있는 동안만 트리거 → **약 4턴 후 스크롤 아웃**. 상태창은 `🔧:🐇🔥`처럼 **이모지**만 출력하지 `!기본모드` 문자열을 재출력하지 않으므로 트리거가 영영 안 돌아온다. 결과: 모드 효과(하드모드=맹수 증가+이혁 겁탈 시도, NTR모드=매 회차 공격, 생존모드=자원 압박, TS모드=캐릭터 코드 m→t 전환)가 컨텍스트에서 사라지고, AI는 모드를 4턴만 기억한 뒤 "무모드" 상태로 흘러간다. **9개 모드는 이 스토리의 정체성 그 자체**인데 엔진 레벨에서 작동 불가.
- **수정안**: 모드 효과를 키워드 로어에서 빼낸다. ① 모드-트리거 로어는 `scan_depth: 1`로 줄여 "유저가 방금 모드를 입력했다"는 감지 전용으로만 쓰고(achat-engine.md 명령어 트리거 권장), ② **모드 효과 정의는 post_history_instructions에 "현재 활성 모드 = 직전 응답 상태창 🔧 이모지를 읽어서 판정"으로 상태 기반 재설계**. 즉 AI가 매 턴 자기 직전 출력의 `🔧:🐇🔥`를 보고 모드 효과를 적용하게 한다(상태창은 매 턴 재출력되므로 절대 안 죽음). ③ 상시 로어 「모드 시스템 정의」(id 1323)는 이미 constant=1이라 매 턴 주입됨 — 여기에 "🔧 이모지 → 모드 효과" 매핑을 명시하면 정의는 살아 있다. 죽는 건 트리거 로어 9개의 "효과" 부분뿐이므로, 트리거 로어는 효과 설명을 제거하고 "이 모드가 입력되면 🔧에 {이모지} 추가" 한 줄로 축약.

### F2. [D / E] 일상어·범용어 키워드 오매칭 🔴
- id 1333 「린」 keys `["린", "황녀", "천황"]` — `린`은 1글자. substring `includes("린")`이 "우**린**", "그**린**", "**린**스", "구원하**린**" 등 무수히 오매칭. 황녀 린이 등장하지 않았는데 매 턴 린 로어가 주입될 수 있음.
- id 1352 「이혁 NTR 공격」 keys `["이혁", "협박", "겁탈", "끌고", "공격"]` — `이혁`은 메인 적대자라 거의 매 턴 등장 → 사실상 상시 로어처럼 동작(의도와 다름). `공격`/`끌고`도 범용어.
- id 1334 「랑랑」 keys `["랑랑", "메스가키", "중국"]` — `중국`은 국가명 범용어. 랑랑 무관 맥락("중국 음식" 등)에 오매칭.
- id 1336 「나타샤」 keys `["나타샤", "인도 무희"]` — `인도 무희`는 자연어 구절(공백 포함). 공백 포함 키는 그 구절이 통째로 등장해야만 매칭 → 사실상 트리거 불가에 가까움.
- **수정안**: 1글자 키 제거(`린` → 키에서 삭제, `황녀`·`천황`만 유지하거나 `린 황녀`/`천황의 금지옥엽`). 범용 국가명 제거(`중국` → 삭제, `랑랑`·`메스가키`만). 자연어 구절 키 분해(`인도 무희` → `나타샤`만, 또는 `무희`). `이혁 NTR 공격` 로어는 키에서 `이혁` 제거하고 `겁탈`·`협박`·`끌고가`로 좁히거나, 차라리 이 로어를 상시 로어로 승격(이혁은 메인 적대자라 효과가 상시 필요).

### F3. [E / D] 28개 키워드 로어 — 항목 수 과다 + 동시 활성 토큰 경쟁
키워드 로어 28개(qa-checklist D "30개 이상 WARN, 50개 초과 FAIL" — 28개는 WARN 경계지만, F1·F2와 복합 시 FAIL). 더 큰 문제는 **동시 매칭**: 베이스캠프 장면에서 유저가 "한지안하고 홍세리한테 가서 이혁 얘기를 물어봤다"를 입력하면 한지안·홍세리·이혁NTR 로어가 동시 매칭 + 모드 로어들 + 환경 로어 → `LORE_TOKEN_BUDGET = 2048` 안에서 priority 순 컷. priority 60대 캐릭터(에반젤린·미카·윤정아·최서진·아야카)는 동시 활성 시 잘려나갈 수 있음.
- **수정안**: ① F1 수정으로 모드 로어 9개가 키워드 풀에서 빠지면 경쟁 완화. ② priority 재정렬 — 메인 4명 관련(한지안·홍세리는 시작 등장이라 80) 유지, 후기 합류 캐릭터는 등장 시점에만 필요하니 priority는 그대로 두되 키를 더 고유하게. ③ 자주 동시 등장하는 캐릭터 로어는 content를 압축(현재 200~360자 → 핵심만 150자 이내).

### F4. [컬처·일관성 / K] 호칭 체계 모순 🔴
첫 메시지에서 이혁이 김현아를 "현아"라 부르고 `{{user}}`를 "달달스무디"라 부른다(`"이혁 | 뭐야, 깨어났네?"` 직전 `"우리끼리는 마저 계산"`). 그런데 호칭 매트릭스(id 1327)는 이혁→{{user}}를 "너/이 새끼/달달스무디(비웃음)"로, 이혁→김현아를 "현아야(친한 척)"로 정의. first_mes 본문에서 이혁이 김현아를 부르는 대사는 없지만 scenario·description은 "현아야"라 함. **더 큰 모순**: `달달스무디`는 {{user}}의 닉네임처럼 쓰이는데, {{user}}는 페르소나 이름으로 치환되는 변수다. 페르소나가 "달달스무디"가 아니면 이혁의 호칭이 캐릭터 설정(특정 별명 비웃음)과 페르소나 이름 사이에서 충돌. first_mes 상태창도 `👥달달스무디|성별`로 박혀 있어 페르소나명 치환과 무관하게 "달달스무디"가 고정 출력될 위험.
- **수정안**: ① first_mes 상태창 `👥달달스무디|성별` → `👥{{user}}|{성별}` (id 1321 상시 로어 양식과 일치시킴). ② 이혁의 "달달스무디" 호칭은 "이혁이 {{user}}를 멸시조로 부르는 비하성 별명을 즉흥적으로 붙인다"는 **방향 지시**로 바꾸거나, first_mes 한정 연출로 명시("이혁은 {{user}}의 이름을 모른 채 비하조 별명을 붙여 부른다"). 호칭 매트릭스도 동일하게 정정.

### F5. [일관성·H / A·D] P0 즉시 행동 순서 ↔ 페이즈 ↔ 모드 — 3중 시스템 충돌
