### F1. [C·H / E·Codex 신규] 이미지 URL 규격 불일치 — description vs 상시 로어 [1302] 🔴 최우선
description은 `![](https://risu.ddsmdy.com/images/%EB%AA%A8%EB%AA%A8/momo/6)` 형식(도메인 risu.ddsmdy.com, 경로 `/images/모모/`, 확장자 없음). 상시 로어 [1302] 「이미지 URL 시스템」은 `![](https://fsion.uk/momo/6.webp)` 형식(도메인 fsion.uk, 경로 없음, `.webp` 확장자). **둘 다 캐시 system block에 들어간다** — block 2(description) + block 3(상시 로어). 모델은 같은 시스템의 두 규격을 동시에 보고 매 턴 어느 쪽을 쓸지 흔들린다. `routes/images.mjs`는 `/images/:storyName/:charDir/:sceneKey` 라우트로 서빙하며 `risu.ddsmdy.com/images/모모/momo/6`은 실제 200 OK, `fsion.uk/...webp`는 외부 도메인이라 깨진다(또는 차단). 라이브 DB에 957장(momo 244/momo2 225/momo3 244/momo4 244) 등록 확인.
- **수정안**: 로어 [1302]의 URL 규격을 description과 동일하게(`risu.ddsmdy.com/images/모모/{code}/{num}`, 확장자 없음) 통일. 더 나아가 description의 「이미지 시스템」 헤더와 로어 [1302]가 거의 동일 중복이므로 — **이미지 규칙은 로어 [1302] 한 곳만 남기고 description 헤더는 hasImageMapping 트리거 보장용 최소 예시 1줄로 축소**. ### F2. [B / A·Codex] 죽은 페이즈 참조 — 존재하지 않는 로어를 가리킴 🔴 최우선
scenario 마지막: "페이즈는 상시 로어 「페이즈 시스템」 + 「타락도 시스템」 참조". post_history 2단계: "현재 Phase + 타락도 확인 → 직전 응답의 Phase·💀와 전환 조건 충족 여부 점검". 그런데 17개 로어 어디에도 「페이즈 시스템」 로어가 없고, Phase의 정의·명칭·전환 조건도 어느 필드에도 없다. scenario에 P1~P4 명칭만 한 줄("P1 일상의 균열 → P2 비밀의 압박 → P3 선택의 순간 → P4 종속의 심화") 떠 있을 뿐. 모델은 없는 표를 찾다가 임의 해석한다.
- **수정안 (Codex 권고 채택)**: 페이즈를 별도 상시 로어로 추가하지 않는다(상시 10개가 되어 토큰 악화). **페이즈를 「타락도 시스템」 로어에 흡수** — 타락도 단계(💀30/45/60/75/90)와 페이즈(P1~P4)를 한 체계로 통합. scenario·post_history의 「페이즈 시스템」 참조 문구를 「타락도 시스템」으로 교체. 타락도가 곧 전개 단계 축이 된다.

### F3. [E / E·Codex] 「성적 어휘 가이드」 vs NARRATION_RULES Tier 1 "가슴/유방" 충돌

### F4. [B·H / A·Codex] 글로벌 선택지 ①②③ 시스템 미충족
