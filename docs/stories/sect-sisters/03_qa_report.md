## FAIL (수정 필수)

### F1. [D / E·A] 상시 로어 13개 — 엔진 "상시 4개+ 과다" FAIL + priority 죽은 값 🔴 최우선
상시 로어(`constant:1`)가 13개: [1357 이미지URL] [1358 일반코드매핑] [1359 고유코드매핑] [1360 자주쓰는코드] [1361 INFO박스] [1362 굴복도매트릭스] [1363 호감도굴복도정의] [1364 합의안전장치] [1365 성적어휘] [1366 호칭매트릭스] [1367 명령어] [1368 응답우선순위]. achat-engine.md §5는 "상시 1~2개 권장, 4개+ 과다", qa-checklist.md D는 "상시 로어 4개 이상 FAIL".
- **코드 근거**: `db.mjs:352-356` `getConstantLore` = `ORDER BY insertion_order`. **priority를 안 본다.** 13개 전부 `insertion_order:100`이라 SQLite 정렬이 임의/삽입순. → [1357~1368]에 매겨진 priority 100/95/90은 **정렬에 1도 영향 없는 죽은 값**. `context-builder.mjs:418-419`에서 13개가 통째로 join되어 block 3에 박힘.
- **현재**: 이미지 코드 매핑 [1358](일반 1~90, 90개 코드) + [1359](고유 1~53, 53개 코드) + [1360](자주 쓰는 코드)만 해도 거대. 매 턴 캐시되지만 캐시 무효화 시(유저노트 변경 등은 block4만 무효화하나, block2 description 수정 시 전체 재캐싱) 비용 폭증.
- **수정안**: 상시 로어를 **4개 이하로 통합**한다. 통합안:
  - **[상시1] 이미지 시스템** = [1357]+[1358]+[1359]+[1360] 통합. 코드 매핑은 압축(범위별 1줄: "67~90 표정 / 1~5,82~90 의상·자위 / 6~63 전희·삽입 / 4,18,22,32,41~49 절정" — 개별 90줄 나열 대신).
  - **[상시2] 진행 시스템** = [1361 INFO박스]+[1362 굴복도매트릭스]+[1363 호감도굴복도정의]+[1368 응답우선순위] 통합. [1368]은 [1362]+post_history의 메타-요약이라 거의 중복 → 흡수.
  - **[상시3] 안전·표현 규칙** = [1364 합의안전장치]+[1365 성적어휘]+[1366 호칭매트릭스] 통합.
  - **[상시4] 명령어 시스템** = [1367] 유지(짧음).
  - 통합 후 각 로어에 `insertion_order`를 1/2/3/4로 부여해 정렬을 명시(priority가 안 먹으므로 insertion_order로 순서 제어).

### F2. [C·H / E·A] 굴복도 단계 키워드 로어 4개 — substring 매칭 실패 또는 폭주 🔴 최우선
[1377 굴복도 0~30] [1378 31~70] [1379 71~99] [1380 100]은 굴복도 구간별 어조·전환조건을 담은 핵심 로어인데 키워드가 부실하다.
- **코드 근거**: `context-builder.mjs:230-252` `keywordMatch` = 순수 `lower.includes(trimmed.toLowerCase())`. 형태소 분석 없음. `matchLoreByKeyword`(257-267)는 `scan_depth*2` 메시지 + userInput을 join해서 substring 검사.
- **[1379]의 키 `더`**: 1글자. "더 해줘", "더워", "더이상", "그런데" 등 거의 모든 한국어 응답에 `더`가 포함 → **굴복도 71~99 발정 로어가 굴복도 0 상태에서도 상시 트리거** → AI가 초반부터 발정 어조를 system에서 보게 됨. 마찬가지 `흐으`(짧음), `좆`(어휘 로어와 중복 트리거).
- **[1377]의 키 `반항`/`거부`/`분노`, [1378]의 `순응`/`굴복`/`수치`**: 추상 명사. 실제 채팅 본문/입력에 "반항"이라는 글자가 직접 등장하는 일은 드묾(AI는 "노려본다"로 묘사하지 "반항한다"고 안 씀). → **죽은 로어**.
- **수정안**: 굴복도 단계 로어 4개를 **상시 로어 [상시2 진행 시스템]에 흡수**한다 — 굴복도 구간 어조는 "현재 굴복도가 N일 때 적용"이라 키워드 트리거가 아니라 항상 참조돼야 하는 규칙이다. constant로 올리면 키워드 매칭 자체가 불필요. (이러면 F1의 "상시 4개" 예산 안에서 [상시2]가 커지지만, 죽은 키워드 로어 4개를 살리는 것보다 정합적.) 또는 키워드를 유지하려면 각 단계 구간 표기(`💞0~30`,`💞31~70` 등 INFO 박스 표기)를 키로 — 단 INFO 박스는 출력에 매번 나오므로 scan에 잡혀 결국 상시화와 같음. → **상시 흡수가 정답.**

### F3. [이미지 / E·A] 이미지 URL 시스템 — description/상시로어/first_mes 3중 도메인·형식 불일치 🔴
- **현재**:
  - `description` "이미지 시스템" 헤더: 형식1 = `https://risu.ddsmdy.com/images/{스토리명인코딩}/{캐릭터코드}/{상황코드}`, 형식2 = `https://n.boniee.uk/BABE/2/GT/{번호}.png`
  - 상시 로어 [1357] "이미지 URL 시스템": 형식1 = `https://boniee.uk/BABE/2/{인물코드}/{번호}.png`, 형식2 = `https://n.boniee.uk/BABE/2/GT/{번호}.png`
  - `first_mes` 실제 박힌 URL: `https://risu.ddsmdy.com/images/%EC%84%B9%ED%8A%B8.../ES/82`
  - → description과 [1357]의 형식1이 **완전히 다른 도메인·경로 구조**. AI는 둘 다 system에서 보고 어느 쪽으로 호출할지 모름.
- **코드 근거**: `context-builder.mjs:360` `hasImageMapping = /!\[.*?\]\(https?:\/\//.test(rawDescription)`. description에 `![](https://...)` 패턴(예시 URL)이 있으면 자동 이미지 카탈로그 주입을 스킵 → 외부 URL 방식으로 동작. 즉 AI는 description+로어의 텍스트 설명에만 의존해 URL을 직접 조립해야 하는데, 그 설명이 3중으로 충돌.
- **수정안**: **하나의 도메인·형식으로 통일.** first_mes에 실제 박힌 게 `risu.ddsmdy.com/images/{스토리명}/{인물코드}/{상황코드}`(확장자 없음)이고 이게 실제 작동 중인 URL이므로 이걸 정본으로:
  - description "이미지 시스템" 헤더 = 2~3줄로 축소(형식1·형식2 예시 URL만 — `hasImageMapping=true` 트리거 보장 목적). 상세 규칙 제거.
  - 상시 로어 [상시1 이미지 시스템] = `risu.ddsmdy.com/images/...` 형식으로 통일. `boniee.uk/BABE/2/` 표기 전부 제거.
  - 형식2(고유 코드)도 실제 작동 URL 확인 필요 — first_mes/description 예시는 `n.boniee.uk/BABE/2/GT/{번호}.png`. 이건 그대로 두되 description·로어 양쪽 동일하게.

### F4. [C / E] `!인스타` 명령어 트리거 로어 scan_depth=4 — 4턴 잔류 재트리거
[1376 인스타 출력]은 명령어 트리거 로어(`!인스타` 입력 시 활성)인데 `scan_depth:4`.
- **코드 근거**: `context-builder.mjs:259-262` — `scan_depth*2`개 최근 메시지를 scan. depth=4면 최근 8개 메시지를 본다. 유저가 한 번 `!인스타 고은서`를 치면 그 후 4턴(메시지 8개) 동안 계속 매칭되어 인스타 로어가 system에 박힘. achat-engine.md §5는 "명령어 트리거 scan_depth:1 권장".
- **수정안**: [1376] `scan_depth`를 **1**로. (그래야 직전 입력에 `!인스타`가 있을 때만 활성.)

### F5. [H / A·E] 시뮬레이션 — 굴복도 단계 로어 미트리거의 복합 결과
F2의 결과로 굴복도 단계별 어조 가이드가 키워드 매칭 실패 시 system에 안 들어옴. AI는 굴복도 단계 어조를 description의 "굴복도 단계별 변화" 섹션 + post_history 기억에만 의존 → 장기 세션에서 굴복도 71+ 인데 어조가 31~70 수준에 머무는 드리프트. 또 [1361] INFO 박스의 "비활성 캐릭터는 마지막 상태 유지"가 모호 — 은서 단독 분기 첫 진입 시 은비 INFO에 무엇을 표시할지(초기값? 생략?) 예시 없음.
- **수정안**: F1·F2 수정(굴복도 단계 로어 상시 흡수) 시 동반 해소. INFO 박스 로어에 "첫 진입 시 미등장 캐릭터는 `[고은비:🩷-100|💞0|👚흰 반팔티+돌핀팬츠|자기 방]` 초기값으로 표시" 예시 1줄 추가.
